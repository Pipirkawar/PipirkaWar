На `record_active(*, tg_user_id)` запускается атомарный MULTI/EXEC-pipeline `ZADD key {member: score} EXPIRE key 172800`: member = `str(tg_user_id)`, score = `clock.now().timestamp()` (unix-ts даёт zero-cost-расширение в `ZRANGEBYSCORE` time-range-аналитику). TTL 48 ч (`_TTL_SECONDS = 172_800`) — sliding-window-renewal на каждом `record_active`; semantically воспроизводит lazy-reset `InMemoryDauCounter` 1-в-1 (после полуночи МСК — новый key, ZCARD по нему =0, старый ещё 48h доступен для cron-снапшотов «вчерашний DAU»). Зачем MULTI/EXEC, а не Lua (как в 4.1-H): для DAU-семантики conditional-логика не нужна — ZADD сам идемпотентен по member-у, EXPIRE no-op-ит если key исчез (теоретически невозможно в single-threaded Redis-execution). >
> **Архитектурное решение (отход от изначального предложения чек-листа):** вместо `LIST + SET (lobby:members)` использовать `LIST + HASH (lobby:enqueued_at)`. SET даёт только membership; HASH покрывает membership (`HEXISTS`) **и** хранит `enqueued_at` per-duel — это нужно для реконструкции `LobbyEntry(duel_id, enqueued_at)` в `pop_oldest`. С SET-вариантом пришлось бы либо запихивать `enqueued_at` в LIST-payload как composite `duel_id|iso` (тогда `LREM` в `remove(duel_id)` потребует O(N)-скан через `LRANGE`), либо вести параллельный HASH — то же самое решение, но с лишним SET-key-ом. Single HASH чище. Atomicity сохраняется (HASH-операции single-threaded в Redis-е; Lua объединяет HEXISTS+HSET+LPUSH в одну атомарную команду).

## 🛑 Известные блокеры / открытые вопросы PR-а

- **Flaky `test_invalid_payload_logs_machine_readable_reason` в `tests/unit/bot/handlers/test_roulette_paid.py` (закрыт в F.1 на main).** В `make ci` ~1% флейков, в изоляции (`pytest <file>::<test>`) — стабильно зелён. Тот же класс проблемы уже был решён в `tests/unit/domain/balance/test_config.py::test_total_above_contract_limit_warns` (см. там комментарий «Патчим logger напрямую, чтобы не зависеть от состояния propagate-цепочки»). PR #133 CI красный по этой же причине, мерж был выполнен принудительно. F.1 закрывает.
- **`SandboxTonConnectVerifier` (4.1-D backlog).** Закрыт в 4.1-F (на main = `555a5c5`): production-имплементация `TonConnectProductionVerifier` в коде; включается через `BOT_TON_CONNECT_VERIFIER_MODE=production`.
