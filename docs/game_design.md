## 0. Политика разработки (обязательно)

> Эти требования имеют приоритет над любыми соображениями скорости/удобства. Это политика компании и одна из главных инструкций для ИИ-разработчиков и людей-разработчиков, работающих над проектом.

### 0.1 Принципы кода

Проект должен быть **эталонным с точки зрения ООП и SOLID**. Каждое архитектурное решение и каждый PR должны быть проверяемы по этим критериям:

| Принцип | Что значит для проекта |
|---|---|
| **S — Single Responsibility** | Один класс/модуль = одна причина для изменения. Хэндлеры aiogram только парсят апдейт; вся бизнес-логика — в `core/`. |
| **O — Open/Closed** | Новые активности (рейды, караваны, ивенты) добавляются через расширение базовых интерфейсов, без правок старого кода. |
| **L — Liskov Substitution** | Любая реализация интерфейса (например, `ICache`, `IPaymentProvider`) должна быть взаимозаменяемой без изменения вызывающего кода. |
| **I — Interface Segregation** | Узкие, целевые интерфейсы. Никаких «толстых» абстракций «на все случаи жизни». |
| **D — Dependency Inversion** | `core/` зависит только от абстракций. Все конкретные реализации (БД, кэш, Telegram API, платёжные провайдеры) внедряются через DI. |

Дополнительно обязательно:

- **Encapsulation** — внутреннее состояние игрока/племени/каравана меняется только через публичные методы доменных объектов; никаких «прямых правок полей» из хэндлеров.
- **Composition over inheritance** — наследование только когда оправдано (`is-a`); во всех остальных случаях — композиция и стратегии.
- **DRY / YAGNI / KISS** — не плодить дубли, не добавлять «на всякий случай», предпочитать простые решения.
- **Чистая архитектура** — слои `domain → application → infrastructure → bot`, направление зависимостей строго внутрь.
- **Тип-чек обязателен** — `mypy --strict` (или эквивалент) для всего `core/`. Никаких `Any`, `getattr`/`setattr` для обхода типов.

### 0.2 Безопасность и целостность данных (приоритет №1)

Безопасность и целостность данных — **главный нефункциональный приоритет**. Любая фича, которая угрожает им, не принимается, даже если работает.

| Область | Требования |
|---|---|
| **Транзакции** | Все операции, меняющие длину/толщину/инвентарь/членство в племенах — в транзакциях БД. Для критичных операций — уровень изоляции `SERIALIZABLE` или явные блокировки `SELECT ... FOR UPDATE`. |
| **Идемпотентность** | Каждая «опасная» операция (начисление длины, выдача предмета, награда за караван/рейд, платёж) имеет idempotency-key. Повторное выполнение безопасно. |
| **Audit log** | Любое изменение длины/толщины/предметов/платежей пишется в неизменяемую таблицу `audit_log` с причиной, источником и idempotency-key. |
| **Валидация ввода** | Все пользовательские данные валидируются через pydantic-схемы на границе bot ↔ core. Никаких сырых строк глубже хэндлера. |
| **Authz** | Каждая команда проверяет: уровень, длину, членство в племени, «занят ли активностью». Проверки вынесены в декораторы/middleware и покрыты тестами. |
| **Анти-race-conditions** | Activity Lock + транзакции; невозможно одновременно начать два похода / получить две награды за один караван / влиять на чужой профиль. |
| **Секреты** | Только из переменных окружения / vault. Никогда не коммитятся, не логируются, не уходят в сообщения пользователю. |
| **Платежи** | Чек/инвойс хранится в БД до начисления; начисление — только после успешного подтверждения провайдера; idempotency по `payment_id`. |
| **Анти-фрод** | Лимиты команд, детекция накруток рефералов, проверка дублей tg_id, ограничение на количество племён у одного игрока. |
| **Резервные копии** | Ежесуточный дамп managed PostgreSQL + point-in-time recovery (если есть). Тест восстановления — раз в квартал. |
| **PII** | Хранится минимум: `tg_id`, `language_code`, никнейм TG только в кэше отображения. Запрос на удаление — поддерживается. |
| **Зависимости** | Регулярный аудит (`pip-audit`/`safety`) и обновление до патч-версий. CI блокирует high/critical уязвимости. |

### 0.3 Качество и процессы

- Каждая задача завершается **тестами**: unit + integration + (по возможности) e2e.
- Покрытие `core/*` тестами — **≥ 80 %** (для MVP было 70 %, политика повышена).
- Code review обязателен. Никаких прямых пушей в `main`.
- CI должен проходить: `ruff`, `mypy --strict`, `pytest`, `pip-audit`.
- Любое отклонение от §0.1/§0.2 фиксируется как технический долг с дедлайном.

## 1. Концепция

**Пипирик Варс** — PvP/PvE чат-игра для Telegram, где игроки растят своего «пипирика», сражаются друг с другом, ходят в походы, водят караваны, участвуют в рейдах и объединяются в племена.

### 1.1 Регистрация игрока и племени

| Сущность | Как добавляется |
|---|---|
| **Игрок** | Пользователь сам обращается к боту в **личных сообщениях** (`/start`). Никаких других способов нет. Если бот видит юзера в группе, но юзер не писал боту в ЛС — играть нельзя. |
| **Племя (= группа TG)** | Любой администратор группы **добавляет бота в группу/супергруппу** и даёт ему права на чтение сообщений и отправку. Бот сам создаёт запись `clans` по `tg_chat_id` и пишет в чат «привет, я зарегистрировал племя…». |
| **Племя (= канал)** | **Не поддерживается** — чтения/реакций пользователей в каналах недостаточно для геймплея, и геймдиз решил отказаться от этой идеи полностью. **Канал-анонсы** (отдельный публичный канал бота с итогами недели / релиз-нотами / лидербордами) — другая фича, она заведена в `current_tasks.md` как backlog-пункт **«канал-анонсы перед публичным релизом»** (отдельный спринт в самом конце Фазы 4). |
| **Привязка игрока к племени** | Игрок становится членом племени автоматически, как только бот видит его сообщения в чате племени **И** игрок зарегистрирован в боте через ЛС (см. выше). До регистрации в ЛС бот его игнорирует и шлёт инструкцию «напишите мне в личку, чтобы вступить». |

> **Важно:** племя **нельзя** «зарегистрировать» через ЛС бота. Это сделано специально — племя = группа в TG, поэтому единственный способ его создать — добавить бота в группу. Это исключает фейковые племена.

> Бот корректно обрабатывает: смену названия группы, превращение группы в супергруппу (новый `chat_id`), удаление бота из группы (племя **замораживается** — статус `frozen`, не удаляется).

#### Стартовые параметры новичка

| Параметр | Значение |
|---|---|
| **Длина** | 2 см |
| **Толщина** | 1 |
| **Титул** | _нет_ (первый титул «Новичок» выдаётся при первом походе в лес) |
| **Название** | определяется по длине автоматически (см. §2.3) |
| **Имя** | _нет_ (выбивается дропом из леса как предмет — см. §2.5) |

> При регистрации новичок отображается без титула и без имени, например: «Пипирик» (только название). После первого леса станет, к примеру: «Новичок Пипирик Иванушка».

### 2.1 Формат имени (единый, везде)

Во всех сообщениях, логах, топах, боях — имя игрока отображается в формате:


Примеры:

- **Нежный Пипирик Иванушка**
- **Толстяк Батон по ГОСТу Геральд**

> Титул — за действия (выбирается), название зависит от длины (автоматическое), имя — выбивается в походах.

### 2.3 Название (автоматическое, по длине)

**Полная таблица «диапазон длины → название»** хранится отдельно и редактируется через админ-панель. На уровне ГДД фиксируется только сам факт того, что:

- Каждой длине соответствует ровно одно название.
- Диапазоны не пересекаются и покрывают **всю числовую ось** (от 0 до «бесконечности», в т. ч. отрицательные значения теоретически — но на практике игрок не может уйти ниже 1 см).
- Изменение таблицы — **горячее**, без перезапуска бота.

**Файл-источник истины:** `config/balance.yaml`, секция `display_names`. Формат:

```yaml
# Полуоткрытый интервал: from <= length < to. Последний to = null = «до бесконечности».
display_names:
  - { from: 0,    to: 10,   name: "Пипирик" }
  - { from: 10,   to: 30,   name: "Писюнчик" }
  - { from: 30,   to: 60,   name: "Бананчик" }
  - { from: 60,   to: 100,  name: "Дубинка" }
  - { from: 100,  to: 200,  name: "Жезл" }
  - { from: 200,  to: 500,  name: "Батон по ГОСТу" }
  - { from: 500,  to: null, name: "Батон-переросток" }  # TODO(gd): полная пересборка
```

> **Текущая таблица — временная и подлежит полной перезаписи.** Геймдиз планирует переработать как сами названия, так и границы диапазонов (включая 501+ см и более ранние). До предоставления финального списка используются значения-заглушки выше.
>
> **Валидация:** при загрузке `balance.yaml` бот проверяет, что диапазоны (полуоткрытые: `from ≤ length < to`) покрывают всю шкалу без дыр и пересечений. Если нет — отказ загрузки и алёрт в админ-чат.

### 2.4 Титул (за действия)

| Ласковый | Участвовать в 5 масс PvP |
| Ядрёный | Победить 10 PvP подряд |

### 2.5 Имя

- **Выбивается в походах** (считается предметом). **При регистрации имени нет.**
- Примеры: Иванушка, Коляндр, Геральд, Пипидон, Батонище.
- **Коллекции нет** — одно активное имя.
- Если у игрока нет имени, в формате отображения оно просто отсутствует: «Титул Название» (без третьей части).
- При нахождении нового имени: **заменить текущее** или **выбросить**. Менять можно **каждый раз**, без ограничений.

### 2.6 Экипировка (чистый декор)

**8 слотов** (6 базовых + 2 руки):

| Слот | Примеры |
|---|---|
| 🎩 Шапка | Шлем Берсерка, Панамка Новичка, Корона Батона |
| 👕 Тело | Кольчуга Лесника, Майка-Алкоголичка, Бронежилет |
| 👖 Штаны | Порты Бывалого, Шорты Дачника, Латные поножи |
| 👟 Обувь | Лапти Скорохода, Кроссы Бегуна, Сапоги-Скороходы |
| 💍 Кольцо | Перстень Мудрости, Колечко Бабули, Печатка Авторитета |
| ⛓️ Цепь | Цепь Кача, Бусы Шамана, Голда с Рынка |
| 🗡️ Правая рука (оружие) | Меч-Кладенец, Нож Бабули, Биток Подъезда, Двуручник Палыча |
| 🛡️ Левая рука (оружие) | Кинжал Заточки, Кастет Кача, Щит Подъезда, Дрын Сторожа |

- **Чистый декор** — никаких бонусов к статам. Оружие в правой/левой руке так же не влияет на бой (бой идёт через 6 кнопок атаки/блока, см. §7.1); это второй декоративный слой поверх «шмот».
- Редкость с **цветовой маркировкой**:
  - ⬜ **Обычный** (серый/белый)
  - 🟦 **Редкий** (синий)
  - 🟪 **Эпический** (фиолетовый)
  - 🟧 **Легендарный** (оранжевый) *(добавлен под высокие тиры заточек, см. §2.8)*
- Не копится: **надеть** или **выбросить**.
- Все 8 слотов — равноценные источники дропа. Оружие (`right_hand`, `left_hand`) дропается из тех же локаций, что и обычный шмот: лес, горы, данжон, награды каравана/рейда, рулетки. Веса дропа на руки vs шапка/тело/… — в `balance.yaml` (`drop.slots`).
- Категории для системы заточки (см. §2.8):
  - **Оружие** — `right_hand`, `left_hand`. Точатся обычными «оружейными» скроллами.
  - **Броня** — `head`, `body`, `legs`, `feet`. Точатся обычными «бронёвыми» скроллами.
  - **Бижутерия** — `ring`, `chain`. Точатся обычными «бижутерными» скроллами.

### 2.7 Основные статы

| Стат | Роль | Описание |
|---|---|---|
| **Длина** (см) | HP / ресурс | Теряется в боях, тратится на прокачку. Набирается в походах, рейдах, караванах, донатах, предсказаниях. |
| **Толщина** | Уровень | Определяет доступные активности. Прокачивается за длину. Без ограничений. |

### 2.8 Заточка предметов

> **Цель механики:** забавный sink для лишних СМ + scroll-фарминговая активность поверх шмот-системы. Заточка **никак не влияет на бой** (как и сама экипировка) — она чисто статусно-косметический показатель.

#### 2.8.1 Что точится

Точатся **все 8 слотов** экипировки (см. §2.6), сгруппированные в три категории:

| Категория | Слоты | Скролл-категория |
|---|---|---|
| **Оружие** | `right_hand`, `left_hand` | `weapon_scroll` |
| **Броня** | `head`, `body`, `legs`, `feet` | `armor_scroll` |
| **Бижутерия** | `ring`, `chain` | `jewelry_scroll` |

Скролл одной категории нельзя применить на предмет другой категории (например, `weapon_scroll` на шапку — отказ с подсказкой). Скроллы — отдельная сущность инвентаря, **копятся** (стакаются), не имеют слотов «надевания»; используются только через действие «заточить».

#### 2.8.2 Лестница заточки и тиры сложности

Уровень заточки — целое от **+0** до **+30**. Каждое успешное использование скролла повышает уровень на `+1` (обычный скролл) или `+1`/`+2` (blessed). Каждое неудачное использование — снижает или разрушает (см. §2.8.3, §2.8.4).

Тиры сложности задают **качественную сложность** заточки в диапазонах. Точные числовые вероятности — в `balance.yaml` (см. §2.8.6). Финальные значения подбираются балансом после фактических метрик игроков, поэтому в ГДД фиксируется только направление:

| Тир | Диапазон | Описание сложности | Качественная вероятность успеха обычной заточки |
|---|---|---|---|
| 🟢 Безопасный | **+0 → +3** | Гарантированно успешно. Никакого риска. | **100 %** успех; падений и разрушений нет. |
| 🟢 Лёгкий | **+3 → +7** | Чаще успех, чем неудача. Большинство игроков легко достигают. | Высокий шанс успеха (≈ 70–90 %). |
| 🟡 Сложный | **+7 → +12** | Половина или меньше — успех. | Средний шанс (≈ 40–60 %). |
| 🟠 Очень сложный | **+12 → +18** | Чаще неудача, чем успех. | Низкий шанс (≈ 15–35 %). |
| 🔴 Чрезвычайно сложный | **+18 → +25** | Большинство попыток — провал. | Очень низкий шанс (≈ 5–15 %). |
| ⚫ Почти нереальный | **+25 → +30** | Hall-of-fame достижение. | Околонулевой шанс (≈ 0.5–3 %). |

> Конкретные `success_probability_by_level` для каждого тира и каждого уровня внутри тира — определяются балансом через `balance.yaml`. Описанные выше диапазоны — просто **горизонтальные ориентиры**, какие приблизительно числа должны попадать в каждый тир.

#### 2.8.3 Обычные скроллы — исходы

При применении **обычного** скролла одной из категорий на соответствующий предмет, ролится один из 4-х исходов (взвешенно, веса в `balance.yaml`):

| Исход | Эффект |
|---|---|
| ✅ **Успех** | Уровень предмета повышается на `+1`. Игрок видит «+N → +N+1». |
| ⚪ **Нет эффекта** | Скролл потрачен, уровень не меняется. Игрок видит «без изменений». |
| 🔻 **Падение** | Уровень опускается **на 3** (но не ниже `+0`). Игрок видит «+N → +max(0, N-3)». |
| 💥 **Разрушение** | Предмет **навсегда удаляется из инвентаря**. Игрок видит «предмет разрушен». |

**Игроку перед применением скролла обязательно показывается предупреждение** с явным текстом:

> ⚠️ **Внимание!** Заточка обычным скроллом может **сломать предмет навсегда**. Текущий уровень: **+N**. Подтвердить?

Кнопка **«Подтвердить»** — конкретно эта кнопка инициирует ролл; без подтверждения скролл не расходуется.

В тире `+0 → +3` исходы «падение» и «разрушение» **отключены** (вес 0). Это «безопасная зона»: только «успех» и (опционально, если так выпадет в `balance.yaml`) «нет эффекта», но обычно — **всегда успех**.

#### 2.8.4 Blessed-скроллы — исходы

При применении **blessed** (благословлённого) скролла:

| Исход | Эффект |
|---|---|
| ✅ **Успех малый** | Уровень повышается на `+1`. |
| ✅ **Успех двойной** | Уровень повышается на `+2`. |
| ⚪ **Нет эффекта** | Скролл потрачен, уровень не меняется. |
| 🔻 **Падение малое** | Уровень опускается **на 1** (но не ниже `+0`). |
| 🔻 **Падение двойное** | Уровень опускается **на 2** (но не ниже `+0`). |

**Blessed-скролл НЕ ломает предмет.** Это его ключевое отличие от обычного и оправдание более редкого дропа.

Перед применением — предупреждение:

> ⚠️ **Внимание!** Благословлённая заточка **никогда не разрушит предмет**, но при неудаче уровень может **снизиться на 1 или 2**. Текущий уровень: **+N**. Подтвердить?

В тире `+0 → +3` blessed работает аналогично обычному: только «успех малый» (всегда `+1`); «двойной успех», «нет эффекта», «падения» — отключены.

##### Граничные правила: близость к `+30`

`max_level = +30` — захардкоженный потолок. У границы — два специальных правила:

| Уровень предмета | Что разрешено |
|---|---|
| **+29** (одно очко до потолка) | Blessed: только `success_1` (+1, итог +30), `no_effect`, `drop_1` (-1), `drop_2` (-2). **Исход `success_2` (+2 → +31) запрещён** — он повёл бы предмет за `max_level`. Веса `success_2` на уровне +29 в `balance.yaml` обязаны быть `0.0`. |
| **+30** (потолок достигнут) | Заточка **невозможна**. Любая попытка применить скролл на предмет с `enchant_level == max_level` отвергается доменной ошибкой `MaxLevelReached`. Игрок видит: «Предмет уже заточен на +30 — это максимум». Скролл при этом **не расходуется**. |

Это инвариант на уровне домена (use-case `EnchantItem` и picker `pick_enchant_outcome`), плюс инвариант pydantic-схемы конфига (см. §2.8.6). Дублирование специально — defence-in-depth: если кто-то поправит `balance.yaml` мимо валидатора, доменный код всё равно не выпустит предмет за `+30`.

Симметричное правило **не нужно** на нижней границе: исход `drop_2` на уровне +1 даёт `clamp(0, +1 - 2) = 0`, что валидно.

#### 2.8.5 Источники скроллов и редкость дропа

Скроллы дропаются из активностей. Точные числа (вероятности) — в `balance.yaml`; в ГДД фиксируем только **относительный порядок редкости**, чтобы дизайн оставался узнаваемым.

| Источник | Обычный скролл | Blessed-скролл |
|---|---|---|
| ⛰️ Горы (`/mountains`) | Очень-очень малый шанс | — |
| 🏰 Данжон (`/dungeon`) | Очень малый шанс | Очень-очень малый шанс |
| 🐉 Победа над рейд-боссом | Малый шанс (per-player ролл, не «один скролл на пати») | Очень малый шанс (per-player) |
| 🎰 Free-рулетка (см. §12.4) | Низко-средний шанс | Очень малый шанс |
| 🎰 Платная рулетка (см. §12.5) | Средний шанс | Низкий шанс |

> Лес и предсказатель **не дропают скроллы** (это безопасные зоны для новичков, см. §3.1; sink-механика начинает работать с гор/данжонов и выше). Из каравана скроллы тоже не дропаются — там доминирует «длина» как основной приз.

Per-player ролл на боссе значит: каждый из участников рейд-пати, который пережил бой, получает **свой собственный** независимый ролл шанса дропа. Не «один скролл на всех» — это даёт правильный стимул собираться в рейды.

#### 2.8.8 Экономика и sink-эффект

- **Цель механики** — не «дать игроку преимущество», а **вывести лишние СМ**. Игроку с длиной 50 000 см и 100+ скроллами в инвентаре есть «куда вкладываться»; разрушения возвращают часть СМ обратно в систему (приобретение новой шмотки → лес/каравану/рейд → ещё СМ).
- Поскольку заточка **не влияет на бой** (см. §7), баланс защищён по построению: даже игрок с +30 на всех слотах не получает преимущества в дуэли. Единственные «эффекты» — статус, цветовая разметка, лидерборд «топ-10 заточек» (опциональный, см. §14).
- **Анти-чит** для системы заточек: каждый ролл проходит через стандартный `IRandom` (не `random` напрямую), `IIdempotencyKey`-защищённый use-case, audit `ITEM_ENCHANT_ATTEMPT` пишется в `audit_log` с `before`/`after` уровнями. Подделать favorable роллы напрямую через прямую БД-правку — детектится trip-wire-rope (например, аномальная серия `+1` подряд → admin alert; см. §3.3).

## 3. Минимальная длина (порог безопасности)

### 3.1 Правило 20 см

Если у игрока длина **< 20 см**, он **не может** участвовать в активностях, где есть риск потери длины.

| Активность | Может давать −длину? | Требует ≥ 20 см? |
|---|---|---|
| 🌲 Лес | ❌ Нет, **всегда +** | ❌ Нет, доступен всегда |
| 🔮 Предсказание | ❌ Нет, **всегда +** | ❌ Нет, доступно всегда |
| ⛰️ Горы | ✅ Да | ✅ Да |
| 🏰 Данжон | ✅ Да | ✅ Да |
| ⚔️ PvP 1×1 | ✅ Да | ✅ Да |
| ⚔️ Масс PvP | ✅ Да | ✅ Да |
| 🐪 Караван (караванщик) | ✅ Да | ✅ Да, причём **после вычета взноса** ≥ 20 см |
| 🐪 Караван (защитник) | ✅ Да | ✅ Да |
| 🐪 Караван (рейдер, ур. 5+) | ✅ Да | ✅ Да |
| 🐉 Рейд-босс | ✅ Да | ✅ Да |

> **Караван:** Если взнос 15 см, а у игрока 34 см, то после вычета останется 19 см (< 20) — **не может** присоединиться. Нужно минимум 35 см.

### 3.2 Безопасная зона

Игрок с < 20 см **всегда может**:
- Ходить в **лес** (всегда даёт + длину).
- Использовать **предсказание** (всегда даёт + длину).

> **Прокачка толщины** тоже требует, чтобы после вычета стоимости осталось **≥ 20 см**. Например: у игрока 50 см, прокачка стоит 35 см → останется 15 см (< 20) → **нельзя**. Нужно минимум 55 см.

Это гарантирует, что игрок **никогда не застрянет** — всегда может набрать длину через лес и предсказания.

### 3.3 Хардкап роста длины (анти-чит)

> **Цель:** защитить экономику от багов, эксплойтов и злонамеренных накруток. Любая прибавка длины из **игровых источников** (organic) проходит через жёсткий лимит. **Донатная длина не имеет лимита** — за деньги доступна любая длина (см. §12 «Монетизация»).

#### 3.3.1 Лимиты

| Окно | Лимит organic-прироста | Что включается |
|---|---|---|
| **Сутки (rolling 24 ч)** | **3 000 см** | Лес, предсказатель, рефералка, бонусы боссов/караванов/PvP-наград, ручные `/grant_length` от админа, любые другие положительные `LENGTH_DELTA` из игровой механики |
| **Неделя (rolling 7 дней)** | **14 000 см** | То же |

> **Не включается в счётчик:** донат (Stars / TON / USDT). Донатная длина пишется в `audit_log` с `source = "donate"` и **полностью игнорируется** агрегацией хардкапа. Игрок может купить любое количество см без ограничений.
>
> **Почему rolling, а не календарные сутки:** rolling-окно (`ts > now() - INTERVAL '24h'`) защищает от обхода через «прибавил 2999 в 23:59 → ещё 2999 в 00:01 → 6000 за полчаса». Календарный сброс в полночь МСК (как у `/oracle`) даёт зону уязвимости в районе границы суток.

#### 3.3.2 Поведение при приближении к лимиту (soft-cap, штатный путь)

Перед каждым начислением длины use-case `progression.add_length(...)` проверяет:

```
remaining_daily  = 3000  - sum_organic_last_24h(player_id)
remaining_weekly = 14000 - sum_organic_last_7d(player_id)
allowed_delta    = max(0, min(delta_requested, remaining_daily, remaining_weekly))
```

- Если `allowed_delta < delta_requested` — начисление **обрезается** (clamp). Игрок получает столько, сколько влезает в остаток окна. В `audit_log` пишется фактическое начисление + поле `clamped_from = delta_requested`. Игроку показывается сообщение в стиле «🛑 Достигнут суточный/недельный лимит роста (3000/14000 см). Начислено: N см. Следующее окно открывается через X часов».
- Если `allowed_delta == 0` — начисление **полностью пропущено**. В `audit_log` запись с `delta = 0`, `clamped_from = delta_requested`, `reason_extra = "daily_cap"` или `"weekly_cap"`. Активность (например, поход в лес) при этом считается завершённой и Activity Lock снимается — **игрок не зацикливается** в попытке получить длину.

#### 3.3.3 Поведение при превышении лимита (hard-trigger, аномалия)

Если каким-то образом (баг, race-condition, эксплойт, прямая БД-правка) **фактическая** rolling-сумма organic-прироста за последние 24 ч у игрока **превысила 3 000 см** или за 7 дней — **14 000 см**, срабатывает **анти-чит trip-wire**:

1. **Soft-ban на 14 дней (rolling)** записывается в `players.anticheat_ban_until = now + 14d`. На время бана игрок:
   - **не может расти** — `progression.add_length(...)` вернёт ошибку `AnticheatSoftBanError`, активности (`/forest`, `/oracle`, рейды, караваны, реферальные бонусы) на время бана не дают длины (их можно либо запретить вовсе, либо начислять 0 см и писать `clamped_from`).
   - **не может тратить длину** на прокачку толщины и PvP (`/upgrade`, `/duel`, масс-PvP, рейды) — все спендалки бракуют ход.
   - **может** общаться, использовать `/profile`, `/top`, `/help`. Бан изолирует прогрессию, не игрока.
2. **Запись в `audit_log`** с `action = "ANTICHEAT_DAILY_CAP_EXCEEDED"` или `"ANTICHEAT_WEEKLY_CAP_EXCEEDED"`, payload содержит:
   - снимок rolling-суммы (сколько фактически прибавилось)
   - последние 50 `LENGTH_DELTA`-записей за окно (для разбора)
   - триггер-источник (use-case, который последним сработал)
3. **Алерт админам** — отправляется в админ-группу: `⚠️ Анти-чит: игрок [<тег>] достиг суточного хардкапа (X см / 3000), soft-ban до <дата>. Audit-id: <uuid>.` Алерт идемпотентен — за один и тот же триггер-id повторно не шлётся.
4. **Бан снимается автоматически** через 14 дней (по таймеру) ИЛИ вручную админом командой `/anticheat_unban <tg_id> <reason>` (только `super_admin`, обязательная причина, запись в `admin_audit_log`).

> **Почему 2 линии (clamp + trip-wire), а не одна:** `clamp` защищает от штатных багов (округление, +1 от рефералки, последовательные mutations). `trip-wire` ловит то, что прорвалось мимо `clamp` — гонки, прямые БД-правки, ошибочные ручные `grant_length`, баги в новых use-cases, забывших позвать `clamp`. Это эшелонированная защита.

#### 3.3.4 Что считается «organic» (источники, попадающие под лимит)

| Источник | Под лимитом? | Причина |
|---|---|---|
| Лес (`/forest`) | ✅ Да | Игровая прогрессия |
| Предсказатель — базовый бонус (`source = "oracle"`) | ✅ Да | Игровая прогрессия |
| Предсказатель — **бонус-за-племена** (`source = "oracle_tribe_bonus"`) | ❌ Нет | **Виральная мини-механика**, отдельный лимит (см. §11.1). Кап `+131 см` за вызов, итого `/predict` ≤ `+151 см`. Аггрегация organic 24h/7d **игнорирует** этот источник — он в blacklist-е `tribe_bonus_sources`, не в whitelist-е `organic_sources`. |
| Рефералка (`+5/+1/+10/+30`) | ✅ Да | Защита от накрутки рефералов |
| Награды PvP / караванов / рейдов | ✅ Да | Игровая прогрессия |
| Админ-команды `/grant_length` | ✅ Да | Защита от ошибок и злоупотреблений админов; записывается с `source = "admin_grant"`, считается в окне |
| **Донат** (Stars / TON / USDT) | ❌ Нет | Платный канал — без ограничений по длине |
| Возврат при админ-сторно | ❌ Нет | `source = "admin_refund"` обнуляет счётчик действия (записывается с отрицательной дельтой и не учитывается в положительной агрегации) |

#### 3.3.5 Конфигурация и hot-reload

Лимиты находятся в `balance.yaml`, секция `anticheat`:

```yaml
anticheat:
  daily_cap_cm: 3000          # rolling 24h
  weekly_cap_cm: 14000        # rolling 7d
  soft_ban_duration_days: 14  # длительность бана при срабатывании trip-wire
  organic_sources:            # whitelist источников, которые считаются в окне
    - forest
    - oracle
    - referral_signup
    - referral_thickness
    - pvp_reward
    - caravan_reward
    - raid_reward
    - admin_grant
  donate_sources:             # blacklist (НЕ считаются)
    - stars_payment
    - ton_payment
    - usdt_payment
```

Меняется без рестарта через `/balance_reload` (см. §0.3 ГДД и Спринт 1.1.E). Все изменения логируются в `admin_audit_log`.

#### 3.3.6 Тест-сценарии (DoD)

- ✅ 10 последовательных успешных лесов суммарно дают ровно `min(сумма_наград, 3000)` см.
- ✅ После достижения 3 000 см за сутки следующий `/forest` возвращает 0 см и сообщение «достигнут суточный лимит».
- ✅ После достижения 14 000 см за неделю следующее любое начисление возвращает 0 см.
- ✅ Донат на 5 000 см в один платёж зачисляется полностью — не блокируется хардкапом.
- ✅ Параллельный запуск 100 завершений `/forest` (race-test) не приводит к превышению `daily_cap_cm` ни на 1 см (тест в `tests/integration/load/`).
- ✅ Если в обход штатных use-cases (через прямой `repo.save(player)` в тесте) длина прибавилась так, что rolling-сумма стала > 3 000 — следующая попытка mutation триггерит trip-wire (`ANTICHEAT_DAILY_CAP_EXCEEDED`, soft-ban на 14 дней, алерт админу).
- ✅ Игрок в soft-ban не может ходить в лес, прокачиваться, дуэлиться, получать рефералку. Может смотреть `/profile`, `/top`, `/help`.
- ✅ `/anticheat_unban <tg_id> <reason>` от `super_admin` снимает бан и пишет `admin_audit_log` с причиной.

## 6. Племена (Группы Telegram)

- Каждая **группа Telegram** = **племя**.
- Игрок может состоять в **нескольких племенах**.
- Если игрок в обоих враждующих племенах — **пропускает** масс PvP.
- **Топ племён** — суммарная длина.

### 6.1 Глава племени дня 👑

Каждые сутки в каждом племени автоматически выбирается случайный «глава племени дня» — для атмосферы и небольшого юмора в чате.

#### Правила:

| Параметр | Значение |
|---|---|
| **Активация** | Племя имеет **≥ 5 активных участников** (зарегистрированных в боте через ЛС). |
| **Лимит частоты** | **Раз в сутки** (по `Europe/Moscow`, как и `/oracle`); сутки сбрасываются в 00:00 МСК. |
| **Триггер — гибридный** (см. ниже) | Кнопка «🎲 Назначить главу дня» в чате племени **ИЛИ** автоматический фоновый розыгрыш в случайное время суток. Что наступит первым — то и побеждает. |
| **Кто выбирается** | Случайный участник племени из активных за последние 7 дней. |
| **Срок полномочий** | До следующего успешного розыгрыша (≤ 24 ч). |
| **Бонус главе дня** | **`uniform(1, 20)` см** длины (балансируемо в `balance.yaml`). Никаких боевых бонусов. |
| **Можно ли выбрать одного и того же** | Можно, но если в племени ≥ 10 участников — алгоритм избегает повтора последних 3 победителей. |
| **Совмещение с другими активностями** | Не блокирует ничего; это «титул дня», а не активность. |

#### Гибридный триггер (важно):

Чтобы не создавать «пиковую нагрузку» в 12:00 МСК (когда тысячи племён одновременно шлют сообщения), розыгрыш реализован двумя способами одновременно:

1. **🎲 Кнопка / команда `/clan_head` в чате племени.** Любой участник может нажать в любое время. Если на сегодня ещё нет назначенного главы — бот тут же выбирает и публикует. Если уже есть — бот тихо ответит подсказкой («сегодня уже назначен, следующий — после полуночи МСК»).
2. **Фоновый APScheduler-cron.** Для каждого племени при создании/активации генерируется детерминированный персональный offset `0..24 ч` (на основе `clan_id` + `date`). В этот момент cron сам тыкает розыгрыш, если кто-то ещё не сделал это через кнопку.

**Идемпотентность:** уникальный ключ `(clan_id, moscow_date)` в `clan_daily_head`. Параллельные «кнопка + cron» не приведут к двойному назначению.

> Эффект: распределённая нагрузка по суткам + игровой элемент «кто первый дёрнет рулетку». Больше нет необходимости синхронизировать всех на 12:00 UTC.

#### Сообщение в чат племени:

Бот публикует **иронично-смешное** сообщение, заканчивающееся «пацанской цитатой» (в стиле Стэтхема / пабликов ВК / АУФ-цитат). Полное имя игрока — в формате «Титул Название Имя».

```
👑 Главой племени "Огурцы" на сегодня волей рандома (и кнопки от @vasya) назначен:
   Ядрёный Бананчик Коляндр

📏 Получает +14 см к длине за тяжёлое бремя власти и моральные страдания.
🕒 Полномочия — до полуночи по Москве. Потом обратно в строй, как все.

«Пацан сказал — пацан подумал… ну, может быть. Главное — племя стоит, а длина растёт.» 🐺
```

#### Каталог цитат:

- На MVP — **JSON-файл** `templates/clan_quotes_ru.json` / `_en.json` с **≥ 100 цитатами**.
- Цитаты — **ироничные и смешные** (а не пафосные/серьёзные). Стилистика — Стэтхем / паблики ВК / АУФ-цитаты, но с обязательной самоиронией.
- **Уместный мат разрешён** (`mild_profanity = true` в контент-полиси). Цитата с матом помечается тегом `profanity` для опционального фильтра «детский режим племени» в будущем.
- **Запрещено:** политика, межнациональные оскорбления, призывы к насилию, реклама (см. `templates/content_policy.md`).
- Каждая цитата отмечена тегами стиля (`statham`, `vk_pablik`, `auf`, `meme`) и опциональным `profanity` — на будущее для A/B-теста и фильтров.
- Позже — генерация ИИ с проверкой на безопасный контент.

#### Технические требования:

- Реализуется через `application/clan/daily_head/*` use-cases:
  - `RequestDailyHead(clan_id, requested_by)` — синхронный, вызывается из бот-кнопки.
  - `RunDailyHeadCron(clan_id)` — фоновый, идёт от APScheduler-job.
  - Оба использует общий доменный сервис `DailyHeadService.assign_or_get(clan_id, source)` с идемпотентностью.
- Запись в `clan_daily_head` (`clan_id`, `moscow_date` (PK с `clan_id`), `user_id`, `quote_id`, `bonus_cm`, `source` ∈ `{button, cron}`, `requested_by` (nullable)).
- **Идемпотентность**: повторный вызов за те же сутки = no-op (PK = `clan_id, moscow_date`); параллельные кнопка+cron не дают двойного назначения.
- Все начисления длины — через стандартный сервис `progression.add_length(reason="daily_head")` с записью в `audit_log`.
- Если бот удалён из племени / племя в `frozen` или `archived` — оба триггера пропускают этот племя.

> Идея: дополнительная мини-фича — глава дня может **раздать** свой бонус другому участнику племени (опционально, в Фазе 2+).

## 7. PvP-система

### 7.1 PvP 1×1 (с уровня 2, длина ≥ 20 см)

#### Система вызовов:

| Тип вызова | Как работает |
|---|---|
| **Чат → Глобал** (по умолчанию) | Вызов в чате с кнопкой «Принять». **Первые 3 мин** — только участники чата. **После 3 мин** — автоматически в глобальное лобби. |
| **Только чат** | Остаётся только в чате. Никто не принял — отмена. |
| **Только глобал** | Сразу в глобальное лобби. В чате не отображается. |

#### Механика боя:
- 6 кнопок: 3 атаки + 3 блока.
- Каждый выбирает **1 атаку + 1 блок** (через ЛС боту).
- **3 раунда**.
- Попал не в блок → защитник теряет длину. В блок → 0.
- Забавные комментарии после каждого раунда.

### 7.2 Масс PvP (Племенные войны) — все уровни, длина ≥ 20 см

- Племя атакует **не чаще 1 раза в 6 часов**.
- Все участники с длиной ≥ 20 см, **независимо от уровня**.
- Игроки в обоих племенах — пропускают.

**Механика N vs M:**
- Каждый выбирает 1 атаку + 1 блок.
- Каждый бьёт 1 случайного противника.
- Попал не в блок → забирает часть длины. В блок → по нулям.

## 8. Походы (PvE)

### 8.1 Локации

| Локация | Уровень | Длина | Кулдаун (рандом) | Результат | Предметы |
|---|---|---|---|---|---|
| 🌲 Лес | 1+ | Любая | 10–20 мин | **Всегда +** длина | 0–1 шт |
| ⛰️ Горы | 3+ | ≥ 20 см | 20–40 мин | **±** длина | 0–1 шт |
| 🏰 Данжон | 6+ | ≥ 20 см | 40–60 мин | **±** длина | 0–3 шт |

### 8.2 Механика

1. Игрок выбирает поход → проверка уровня, длины и кулдауна.
2. **Сообщение в чат:** «🌲 Ядрёный Бананчик Коляндр ушёл в лес на 14 минут...»
3. Кулдаун тикает. Игрок **заблокирован** от других активностей.
4. По возвращении — **новое сообщение**:

```
🌲 Ядрёный Бананчик Коляндр вернулся из леса!
📏 Длина: +5 см (было 47, стало 52)
🎩 Нашёл: Панамка Новичка [обычный]
   → Надеть | Выбросить
```

- Лес — **всегда + длина** (безопасная активность).
- Горы/Данжон — **±** длина (рискованные).
- Имя = предмет, может выпасть.
- Шмот не копится: надеть или выбросить.

#### Прибавка длины в лесу

Лес имеет **3 ветки исходов** с разной частотой и разным диапазоном прибавки. Распределение веса между ветками — в `balance.yaml`. **Все три ветки — положительные** (лес всегда +).

| Ветка | Диапазон (см) | Стартовый вес | Описание (для логов) |
|---|---|---|---|
| 🥬 Скудный лес | **1–10** | 50 % | «Нашёл +N см под кустом» |
| 🌳 Обычный лес | **5–15** | 35 % | «Зацепился за корягу и нашёл +N см» |
| 🌲 Тучный лес | **10–20** | 15 % | «Наткнулся на полянку и поднял +N см» |

```yaml
forest:

    - { name: "scarce",   weight: 50, min: 1,  max: 10 }
```

## 9. Караваны (создание с уровня 7, нападение с уровня 5)

### 9.1 Концепция

Караван — групповая перевозка длины из племени в племя. Высокий риск + высокая награда. На караван могут напасть рейдеры, а племя-получатель может выставить защитников.

### 9.2 Создание каравана

- **Создатель (лидер):** уровень **7+**, длина после вычета взноса **≥ 20 см**.
- Лидер задаёт:
  - **Вклад длины** — сколько см каждый участник вкладывает.
  - **Пункт назначения** — племя-получатель. Ввести вручную или выбрать из **5 рандомных**.
- Караван создаётся **внутри племени** лидера.

### 9.3 Сбор лобби (20 минут)

- Объявление в чате племени: «🐪 Ядрёный Бананчик Коляндр собирает караван в племя "Батоны"! Вклад: 15 см. Присоединиться?»
- Кнопка **«Присоединиться»** — любой участник племени может вступить, если длина после вычета взноса **≥ 20 см**.
- **20 минут** на сбор. Минимум — **1** (лидер).
- **Кулдаун:** племя может создать караван **не более 1 раза в 12 часов**.

### 9.4 Роли и ограничения для игроков в обоих племенах

| Состоит в... | Караванщик | Защитник | Рейдер |
|---|---|---|---|
| Только в племени-отправителе | ✅ | ❌ | ❌ |
| Только в племени-получателе | ❌ | ✅ | ❌ |
| **В обоих племенах** | ✅ | ✅ | ❌ |
| **Ни в одном из племён** | ❌ | ❌ | ✅ |
| В одном из двух племён | — | — | ❌ |

> **Рейдером может быть только тот, кто НЕ состоит ни в племени-отправителе, ни в племени-получателе.**
>
> Игрок в обоих племенах может выбрать **одну** роль: караванщик **или** защитник (не обе одновременно, т.к. блокировка активностей).

### 9.5 Механика каравана

#### Параметры:

| Параметр | Значение |
|---|---|
| Время в пути | **60 минут** |
| Блокировка | Все участники **не могут** участвовать в других активностях |
| Защиты караванщиков | Каждый выбирает **2 блока** (из 3) перед стартом |

#### Рейдеры (нападающие):

- **Уровень 5+**, длина **≥ 20 см**.
- **НЕ состоит** ни в племени-отправителе, ни в племени-получателе.
- Находят караван через **объявление** или **кнопку «Поиск караванов»**.
- Присоединяются **в любое время** в течение 60 минут.
- **Максимум:** ×4 от количества караванщиков (балансируемо).

#### Защитники (племя-получатель):

- Длина **≥ 20 см**. Все уровни.
- Состоит в племени-**получателе**.
- Присоединяются **в любое время** в течение 60 минут.
- Каждый выбирает **1 блок**.
- **Максимум:** ×2 от количества караванщиков.
- Берут **часть ударов на себя**.

#### Боевая механика:

- Каждый рейдер наносит **1 удар** случайному караванщику или защитнику.
- Общее количество ударов = количество рейдеров.
- Макс. ударов = ×4 × караванщиков.

| Ситуация | Результат |
|---|---|
| Рейдер попал **не в блок** | Караванщик/защитник **погибает** + немного **−длины** |
| Рейдер попал **в блок** | Рейдер получает немного **−длины** |

### 9.6 Завершение каравана

#### 🏴‍☠️ Все караванщики и защитники погибли (рейдеры победили):

- Рейдеры **делят всю перевозимую длину поровну** (округление в бо́льшую сторону системой).
- Один случайный рейдер — **Атаман разбойников**: получает **×4 к своей доле** от системы + **титул «Атаман»**.

#### 🐪 Караван дошёл (хотя бы 1 караванщик или защитник выжил):

| Роль | Награда |
|---|---|
| **Лидер каравана** | **×4 к вложенной длине** + **титул «Торговец»** (при 3-м успешном караване) |
| **Караванщики** (остальные) | **×3 к вложенной длине** |
| **Защитники** | **×1 от системы** (фиксированный бонус длины от системы, не от вклада) |
| **Все участники обоих племён** | **+1 см длины** каждому |

> Все множители (×4 лидер, ×3 караванщики, ×1 защитники, ×4 атаман, +1 племени) — **балансируемые**.

### 9.7 Пример

```
Племя "Огурцы" → Караван в племя "Батоны"
  Лидер: Ядрёный Бананчик Коляндр (вклад 20 см)
  Караванщиков: 5 (вклад 20 см каждый = 100 см общий груз)
  Защитников от "Батонов": до 10
  Рейдеров (не из "Огурцов" и не из "Батонов", ур. 5+): до 20

  Если караван дошёл:
    Лидер:         вложил 20 → получил 80 (×4)
    Караванщики:   вложил 20 → получил 60 (×3)
    Защитники:     получили по X см от системы (×1)
    Все "Огурцы" и "Батоны": +1 см каждому

  Если караван разграблен:
    100 см ÷ рейдеры (поровну)
    1 рейдер = Атаман (×4 к своей доле от системы)
```

## 10. Рейд-боссы

### 10.1 Вызов и участие

| Действие | Уровень | Длина |
|---|---|---|
| **Кинуть вызов** | **9+** | ≥ 20 см |
| **Участвовать** | **4+** | ≥ 20 см |

- Босс = случайный из **топ-30**. Кулдаун вызова: **1 раз в 4 часа** (глобальный).

### 10.2 Управление боссом

- Босс = **сам игрок** (если онлайн) или **бот** (автоматически).

### 10.3 Лобби

- **20 минут** сбор. Кнопка **пересылаемая**. Минимум — **1 игрок**.

### 10.4 Механика

| Роль | Кнопки |
|---|---|
| Босс | 3 атаки → бьёт всех |
| Рейдеры | 3 блока → каждый один |

- Раунды **20 сек – 1 мин**. AFK → бот рандомно.

| Ситуация | Результат |
|---|---|
| Босс попал не в блок | Рейдер выбывает + −длина |
| Босс попал в блок | Рейдер остаётся + забирает длину босса |

### 10.5 Завершение

- **Рейдеры победили** (длина босса < 10 см): делят длину + **% от системы**.
- **Босс победил**: забирает длину рейдеров + **% от системы**.

## 11. Предсказатель

- **1 раз в день**. Мгновенное (не блокирует). **Всегда + длина**.
- **Кулдаун** — по московскому времени (UTC+3). Сутки сбрасываются в 00:00 по Москве (т. е. 21:00 UTC).
- **Прибавка длины** — `uniform(1, 20)` см (база) **+ бонус за племена** (см. §11.1). Базовое распределение параметризуется в `balance.yaml`.
- На старте — база 200+ шаблонов. Позже — ИИ.

```yaml
oracle:
  cooldown_tz: "Europe/Moscow"   # сутки сбрасываются в 00:00 по этому TZ
  bonus_min: 1
  bonus_max: 20
  distribution: "uniform"        # uniform | normal | weighted_buckets
```

### 11.1 Бонус за племена 🎯

> **Виральная мини-механика.** Чем в большем числе **активных племён** (групп TG, где установлен бот) состоит игрок — тем больше его суточный прирост от Предсказателя. Цель — мотивировать игроков тащить бота в новые чаты, где он ещё не установлен.
>
> ✅ **Реализовано в Спринте 3.6** (PR #126 «3.6-A: domain + config + use-case + anti-cheat», PR #127 «3.6-B: bot UI + локали»).

#### Формула

```
final_predict_bonus_cm = uniform(bonus_min, bonus_max) + min(n_active_tribes * cm_per_tribe, cap_cm)
```

где:

- `uniform(bonus_min, bonus_max)` — базовый бонус из §11 (по умолчанию `uniform(1, 20)`), без изменений.
- `n_active_tribes` — число племён, в которых **одновременно**:
  - (a) бот установлен и не `frozen`/`archived` (по `clans.status == "active"`);
  - (b) **строго больше 3 участников** в чате племени (`> 3`, т. е. `>= 4`);
  - (c) игрок числится членом этого племени (см. §1.1: членство = бот видел сообщения игрока в чате **И** игрок зарегистрирован в боте через ЛС).
- `cm_per_tribe` — `+1` см за каждое активное племя (default).
- `cap_cm` — кап на бонус-за-племена. По умолчанию **`+131 см`**, т. е. суммарный бонус Предсказателя ≤ **151 см** (`max(uniform) + cap_cm = 20 + 131 = 151`).

#### Параметры

| Параметр | Значение по умолчанию | Обоснование |
|---|---|---|
| **Cap** | `+131 см` за вызов (итого `/predict` ≤ **151 см** вместе с базой) | Защита от абуза «100 мини-чатов» при сохранении высокого потолка для активных распространителей бота. |
| **Min участников в племени** | **`> 3`** (строго больше 3, т. е. `>= 4`) | Стрижёт «фейковые» племена из 1–3 ботов / мульти-аккаунтов. Жёстче порога `>= 5` из §6.1 («Глава племени дня») — намеренно: тут речь о виральной растяжке, а не о социальной активности (для tribe-bonus-а достаточно минимальной плотности чата, не «активный за неделю»). |
| **Активность игрока в племени** | **факт членства** (без свежей активности) | Не требуется писать сообщения за N дней. Достаточно присутствовать в племени и быть зарегистрированным в боте через ЛС. |
| **Снапшот vs live** | **Live в момент `/predict`** | Считается на момент инвокации use-case-а; если за последние сутки игрока добавили в новое племя или из племени ушёл бот — следующее `/predict` это учтёт (в пределах cooldown-а). |
| **Anti-cheat (§3.3)** | **Отдельный лимит** (НЕ входит в organic +3000/сутки и +14000/неделю) | Бонус-за-племена пишется в `audit_log` отдельной проводкой с `source = "oracle_tribe_bonus"`; этот источник в whitelist-е `tribe_bonus_sources`, не в `organic_sources` — см. §3.3.4 и §3.3.5. |
| **Display** | **Явная отдельная строка** в результате `/predict` | Сообщение игроку **обязано** содержать строку `+N см за племена (M активных племён)` рядом с базовой строкой `+uniform(...) см — базовый бонус`. См. UX-пример ниже. |

#### UX (пример)

```
🔮 Предсказание дня:
«Сегодня твоя длина прирастёт через мудрость старейшин и шум новых племён.»

📏 Прирост:
   +14 см — базовый бонус
   +5 см — за племена (5 активных племён)
   ──────────────────
   +19 см — итого

(было 47 см → стало 66 см)
```

Если игрок не состоит ни в одном активном племени, строка `+N см за племена` **скрывается** (показывается только базовая строка), чтобы не подсвечивать «вы ничего не получили за племена». Это снижает фрустрацию у новичков, у которых ещё нет племён. Опционально (по итогам альфа-теста) — добавить хинт «*вступай в новые племена и получи дополнительные см к каждому предсказанию*» под результатом, если `n_active_tribes == 0`.

#### Технические требования

- Реализуется в `application/oracle/*` (use-case `RequestOracle` или его расширение). Доменный слой кланов / племён (`domain/clan/*`, `clans` table) переименованию **не подлежит** — это docs-only-рефакторинг (см. ниже).
- Расчёт `n_active_tribes` — отдельный доменный запрос, например `IClanRepository.count_active_for_player(player_id, *, min_tribe_size: int) -> int`. Делается **внутри** транзакции use-case-а `/predict`, **до** финального `progression.add_length(...)`.
- Оба начисления длины проходят через стандартный `progression.add_length(...)`, **двумя отдельными проводками**:
  1. Базовая часть: `reason = "oracle_base"`, `source = "oracle"` (входит в `organic_sources` whitelist §3.3.5, считается в rolling-окнах).
  2. Бонус-за-племена: `reason = "oracle_tribe_bonus"`, `source = "oracle_tribe_bonus"` (входит в `tribe_bonus_sources` blacklist, **НЕ** входит в `organic_sources`).
- Идемпотентность: тот же idempotency-key `oracle:{player_id}:{moscow_date}` уже защищает от двойного запроса; бонус-за-племена считается только при **первом** успешном вызове за сутки.
- **Rename clan → tribe — только в документации** (этот ГДД, ПД, current_tasks, history). Доменные идентификаторы `domain/clan/*`, `application/clan/*`, table `clans`, локали `clan-*`, команды `/clan`, `/freeze_clan`, `/clantop` — **сохраняются как есть** как намеренный docs-only-рефакторинг (без миграции БД, без переименования файлов). Это снижает риск регрессий и позволяет применить только в одной PR-итерации.

#### Балансовый конфиг

```yaml
oracle:
  cooldown_tz: "Europe/Moscow"
  bonus_min: 1
  bonus_max: 20
  distribution: "uniform"
  tribe_bonus:
    enabled: true            # глобальный feature-flag
    cm_per_tribe: 1          # +1 см за каждое активное племя
    cap_cm: 131              # max бонус-за-племена; итого /predict <= 151 см вместе с базой
    min_tribe_size: 4        # > 3 участников (т. е. >= 4)
```

#### Тест-сценарии (DoD)

- ✅ Игрок в 0 активных племён → бонус-за-племена = `0 см`; строка `+N см за племена` в сообщении **скрыта**.
- ✅ Игрок в 5 активных племён (бот установлен, не `frozen`, > 3 участников, игрок — член) → бонус = `+5 см`; строка показана.
- ✅ Игрок в 200 активных племён → бонус = `+131 см` (cap), не `+200 см`.
- ✅ Племя в `frozen` (бот удалён из чата) → не считается, даже если игрок в нём числится.
- ✅ Племя ровно с 3 участниками → не считается (`min_tribe_size = 4`, требование `> 3`).
- ✅ Племя с 4 участниками — игрок один из них → засчитано (`+1 см`).
- ✅ Игрок **не зарегистрирован в боте через ЛС**, но бот видит его сообщения в чате → **не член** племени, не считается (см. §1.1 «членство»).
- ✅ Игрок зарегистрирован в боте через ЛС, но **не пишет** в данном чате → **не член** племени, не считается.
- ✅ Бонус-за-племена пишется в `audit_log` с `source = "oracle_tribe_bonus"` и **не учитывается** в rolling 24h/7d organic-агрегации §3.3 (тест: 50 предсказаний с +131 см бонуса каждое = `50 × 131 = 6 550 см «в обход» хардкапа` — корректно, так и задумано; trip-wire `ANTICHEAT_DAILY_CAP_EXCEEDED` не срабатывает).
- ✅ Сообщение `/predict` обязано содержать строку `+N см за племена (M активных племён)` при `n_active_tribes > 0`.
- ✅ Live-снапшот: если за последний час игрока добавили в новое племя — следующее `/predict` (если cooldown не истёк, или после сброса в 00:00 МСК) учтёт.
- ✅ Идемпотентность повторного `/predict` в те же сутки: бонус-за-племена не начисляется второй раз (защита по idempotency-key `oracle:{player_id}:{moscow_date}`); audit-проводка `oracle_tribe_bonus` пишется ровно одна.
- ✅ `cm_per_tribe = 0` (через `balance.yaml`-override) → tribe-bonus отключён, базовая логика `/predict` не меняется.

> **Связь с §6.1 «Глава племени дня»:** обе механики используют ту же сущность «племя», но **независимы**. «Глава племени дня» — внутрисистемная социальная активность с цитатой и `uniform(1, 20)` см победителю; tribe-bonus — личный виральный буст к Предсказателю по числу племён. Они могут применяться к одному игроку в один день одновременно (если он стал главой и пошёл к Предсказателю — получит и то, и то).

### 12.4 Free-to-play рулетка

> Каждодневный «дешёвый sink» лишних СМ + первый дешёвый канал к крипто-призам. Барьер входа — низкий (за CM), но и шансы на крупные призы — крошечные.

#### 12.4.1 Стоимость и доступ

- **Стоимость спина** — `100 см`. Списывается через `progression.spend_length(...)` стандартным путём (защита 20 см, audit). Спендалка проходит даже если игрок в активити-локе на лесу/oracle (рулетка — отдельный поток).
- **Кулдауна между спинами нет** — игрок может крутить, пока есть СМ.
- **DAU Gate** не учитывает спины рулетки (это не «новый активный игрок», это уже активный игрок).
- Доступна **с уровня толщины 2** (после первого `/upgrade`), чтобы новички сначала прошли базовый цикл «лес → апгрейд → топ» и не сливали стартовые 20 см на рулетку.

#### 12.4.2 Призы и шансы (стартовые дефолты)

> **Ключевая экономика:** free-рулетка — тот же sink, что и заточка. Цена спина = 100 см, средний CM-выигрыш ≈ **60 см**. То есть за 100 спинов игрок «сливает» примерно `100 × (100 − 60) = 4000 см`. Это и есть забор СМ у игрока. Если средний CM-выигрыш окажется ≥ 100 см — рулетка перестаёт быть sink-ом, чего категорически нельзя допустить (балансовый инвариант, см. §2.8.8).

Стартовые веса исходов (сумма == 1.0):

| Приз | Вес | Заметка |
|---|---|---|
| **+СМ от 10 до 500** | **0.850** | Самый частый исход. Распределение внутри (см. ниже) скошено к малым числам. |
| **Шмот** (включая `roulette_only`-эксклюзивы) | **0.100** | Случайный предмет из dropable-каталога; ~80 % обычные/редкие, ~20 % эпические/легендарные. |
| **Обычный скролл заточки** | **0.040** | Категория (weapon/armor/jewelry) — равновероятна (33.3% / 33.3% / 33.3%). |
| **Blessed-скролл заточки** | **0.005** | Категория равновероятна. |
| **Крипто-приз** (Stars / USDT / TON) | **0.005** | Только если в активном крипто-пуле есть лот соответствующей валюты (см. §12.6). Если пул пуст — этот вес перетекает на «+СМ» (становится 0.855). |

Распределение СМ внутри ветки `+СМ от 10 до 500` (ровно та же sink-логика — баланс смещён к малым):

| Бакет | Диапазон | Вес внутри ветки СМ | Среднее по бакету |
|---|---|---|---|
| мелочь | 10..50 | **0.70** | ≈ 30 см |
| средне | 50..150 | **0.25** | ≈ 100 см |
| хороший | 150..300 | **0.04** | ≈ 225 см |
| 🎉 крупно | 300..500 | **0.01** | ≈ 400 см |

Внутри каждого бакета — `uniform(min, max)`. Ожидаемый CM-выигрыш ветки:

```
E[CM | "+СМ" ветка] = 0.70·30 + 0.25·100 + 0.04·225 + 0.01·400
                    = 21 + 25 + 9 + 4 = 59 см
```

Полное `E[CM | spin]` рулетки (включая «не-СМ» ветки, которые вкладывают 0 см):

```
E[CM | spin] = 0.850 · 59 = 50.15 см
```

Чистый sink за спин: `100 − 50 = ~50 см`. Это **дефолт**, после альфа-теста подбирается окончательно.

```yaml
roulette:
  free:
    cost_cm: 100
    min_thickness: 2          # доступ с уровня толщины 2
    outcomes:
      - { kind: "length",          weight: 0.850 }
      - { kind: "item",            weight: 0.100 }
      - { kind: "scroll_regular",  weight: 0.040 }
      - { kind: "scroll_blessed",  weight: 0.005 }
      - { kind: "crypto_lot",      weight: 0.005 }   # обнуляется, если пул пуст
    length_buckets:
      - { name: "small",  min: 10,  max: 50,  weight: 0.70 }
      - { name: "medium", min: 50,  max: 150, weight: 0.25 }
      - { name: "good",   min: 150, max: 300, weight: 0.04 }
      - { name: "big",    min: 300, max: 500, weight: 0.01 }
```

#### 12.4.3 UX-flow

1. Игрок жмёт `/roulette_free` или нажимает кнопку «🎰 Free-рулетка» в `/profile`.
2. Бот показывает текущую цену (`100 см`), баланс игрока, и вероятностное описание призов («чаще всего — СМ; редко — шмот и скроллы; очень редко — крипто»).
3. Кнопка «Крутить» — подтверждение. После клика:
   - Списывается 100 см через `progression.spend_length(...)`.
   - Ролится приз через `IRouletteEngine.spin(player, kind=FREE)`.
   - Результат добавляется в инвентарь / на длину / в крипто-аккаунт игрока (см. §12.6.4 — выплата).
   - Игроку показывается анимация «крутилки» (textual: серия emoji-сообщений с задержкой 200-300 мс) + финальный результат с аплифтом «уау» для редких призов.
4. Нет «гарантий из коробки» — каждый спин независим. Bad-luck protection (например, «после 100 неудачных спинов гарантирован blessed-скролл») — **не предусмотрена** в MVP; добавим в Фазу 4, если метрики покажут отток.

### 12.5 Платная рулетка (Stars)

> Тот же механизм, но за реальную валюту и с лучшими шансами.

#### 12.5.1 Стоимость и пакеты

| Пакет | Цена | Спинов |
|---|---|---|
| 🌀 Одиночный | **1 ⭐** | 1 |
| 🌀🌀 10-pack | **9 ⭐** | 10 |

- 10-pack — фиксированная скидка 10 % (1 спин в подарок).
- Доступна **со старта** (без ограничения по толщине): игрок может купить и крутить с уровня 1.
- DAU Gate тоже не учитывает (как и free-рулетка).

#### 12.5.2 Содержимое и шансы (стартовые дефолты)

> **Ключевая экономика:** платная рулетка — тоже sink для СМ (помимо своей основной роли — приносить доход проекту). Средний CM-выигрыш на спин ≈ **50 см**, максимум 500 см — **редкость**. Игрок не идёт сюда «за длиной»; основная ценность — повышенный шанс blessed-скролла и крипто-приз.

**Тот же пул призов**, что у free-рулетки (§12.4.2), но веса значимо смещены: меньше «голой длины», больше шмота / скроллов / крипто. Сумма == 1.0:

| Приз | Free-вес | Paid-вес | Δ |
|---|---|---|---|
| **+СМ от 10 до 500** | 0.850 | **0.550** | −0.300 |
| **Шмот** | 0.100 | **0.200** | +0.100 |
| **Обычный скролл** | 0.040 | **0.180** | +0.140 |
| **Blessed-скролл** | 0.005 | **0.050** | **+0.045 (10× выше)** |
| **Крипто-приз** | 0.005 | **0.020** | **+0.015 (4× выше)** |

Распределение СМ внутри ветки `+СМ` (для paid — **ещё сильнее** скошено к малым, чтобы среднее упало с 60 до ~50):

| Бакет | Диапазон | Free-вес | **Paid-вес** | Среднее по бакету |
|---|---|---|---|---|
| мелочь | 10..50 | 0.70 | **0.80** | ≈ 30 см |
| средне | 50..150 | 0.25 | **0.17** | ≈ 100 см |
| хороший | 150..300 | 0.04 | **0.025** | ≈ 225 см |
| 🎉 крупно | 300..500 | 0.01 | **0.005** | ≈ 400 см |

Ожидаемый CM-выигрыш paid-ветки:

```
E[CM | "+СМ" ветка, paid] = 0.80·30 + 0.17·100 + 0.025·225 + 0.005·400
                          = 24 + 17 + 5.625 + 2 = 48.6 см
```

То есть среднее CM-выигрыш ветки `+СМ` в платной — **≈ 49 см**, что ровно «50 в среднем», как ты просил. Полное `E[CM | spin]` paid-рулетки:

```
E[CM | spin, paid] = 0.550 · 48.6 = 26.7 см
```

Заметь: средний CM-выигрыш платной рулетки **меньше**, чем free (26.7 vs 50.15) — потому что в платной 45 % спинов уходит на «не-СМ» призы (шмот / скроллы / крипто), а игрок сюда приходит **не за длиной** — за blessed-скроллами и шансом на крипто. Это by design.

**Ключевые отличия от free-рулетки одной строкой:**
- `+СМ` — реже (0.55 vs 0.85), но в среднем по штуке = `48.6 vs 59` (всё равно меньше).
- **Blessed-скролл — в 10× выше** (0.050 vs 0.005). Это «фишка», ради которой стоит платить ⭐.
- **Крипто-приз — в 4× выше** (0.020 vs 0.005). Но ограничен тем же общим пулом (§12.6) — если пул пуст, и в платной не выпадает.

### 12.6 Крипто-призовой пул

> **Ключевое требование пользователя:** игрок никогда не должен попасть в ситуацию «выиграл, но не могу забрать». Поэтому фунт-распорядитель (`PrizePoolService`) обязан учитывать сетевые комиссии перед публикацией лота.

#### 12.6.1 Источники наполнения пула

- **10 % от каждого донат-зачисления** в Stars / TON / USDT (§12.1-§12.3) — автоматически переводится в пул соответствующей валюты.
- Никаких других источников: пул **не пополняется** из бюджета проекта, не из системы-генератора. Это закрытая re-distribution-петля «доход от китов → призы для всех».

#### 12.6.2 Архитектура пула: отдельные пулы по валютам

Пул — **per-currency**, а не общий:

```
PrizePool {
  stars_pool_balance:   <int>,        # в "звёздах" (Telegram Stars units)
  ton_pool_balance:     <decimal>,    # в нанотонах (1 TON = 1e9 nanoTON)
  usdt_pool_balance:    <decimal>,    # в USDT (decimal с 6 знаками)
}
```

- **Stars** — отдельная валюта, не конвертируется (Telegram Stars не выводится на TON / USDT).
- **TON и USDT** — обе на сети TON, но **раздельные пулы**: кит, задонативший USDT, наполняет USDT-пул; кит, задонативший TON, — TON-пул. Это даёт стимул разнообразить donate-каналы и упрощает логику комиссий.

#### 12.6.3 Лоты: как пул превращается в призы

**Концепция лота:** «лот» — это атомарный приз, который может выпасть в рулетке. Лот имеет валюту, номинал, и зарезервированный буфер на комиссию выплаты.

**Минимальный лот** = `1 USDT-эквивалент + комиссия сети` (TODO(balance): зафиксировать USD-эквивалент для каждой валюты в момент формирования лота через oracle/среднюю цену; для MVP — статический курс в `balance.yaml`).

**Максимальный лот** = `10 USDT-эквивалент`. Если в пуле накоплена сумма больше 10 USDT-эквивалент в одной валюте — формируется один лот на 10 (не больше), остальное **остаётся в пуле**, ждёт следующего цикла.

**Один пул — одна валюта — много лотов.** Если в TON-пуле эквивалент 3 USD — это **3 отдельных лота**, каждый по эквиваленту 1 USD в TON. Это даёт более частые мелкие выигрыши, чем «один редкий большой лот».

**Алгоритм формирования лотов** (run раз в час cron-ом + триггерно после крупного донат-зачисления):

```
def regenerate_lots(pool: PrizePool, fee_estimator: IFeeEstimator):
    for currency in [STARS, TON, USDT]:
        balance = pool.balance_of(currency)
        fee_per_payout = fee_estimator.estimate(currency)
        min_lot_value  = usd_equivalent(1.0, currency)
        max_lot_value  = usd_equivalent(10.0, currency)
        # Каждый лот = (fixed payout) + (fee buffer). Списывается из пула цельно.
        while balance >= (min_lot_value + fee_per_payout):
            lot_value   = min(max_lot_value, balance - fee_per_payout)  # не больше 10 USD-экв
            lot_value   = clamp_to_safe(lot_value, min_lot_value)
            reservation = lot_value + fee_per_payout
            pool.reserve(currency, reservation)
            create_active_lot(currency, lot_value, fee_buffer=fee_per_payout)
            balance -= reservation
```

Лоты добавляются в активный пул `IRouletteEngine`-а; пока в пуле есть хотя бы один активный лот данной валюты — он может выпасть в рулетке. Когда лот выигран — он удаляется из активного пула. Если игрок не претендует на выплату (например, сетевая комиссия выросла после формирования лота) — лот возвращается в пул и пересоздаётся.

#### 12.6.4 Выплата выигрыша игроку

**Сценарий выплаты:**

1. Игрок выиграл лот (`stars=10` / `ton=1.5` / `usdt=2.5`).
2. Бот пишет в чат: «🎉 Вы выиграли 2.5 USDT! Чтобы получить выплату, нажмите «Привязать кошелёк».»
3. Игрок указывает свой TON-адрес (для TON и USDT) или ничего (для Stars — выплата в их Stars-аккаунт через Telegram Bot API `payments.refund`).
4. Бот при подтверждённом адресе:
   - Stars: запрос на refund через Telegram Bot API. Комиссия — 0 (Telegram не берёт).
   - TON / USDT: транзакция на сети TON с использованием `fee_buffer` лота. Если на момент выплаты комиссия выше зарезервированной — **бот не выплачивает**, кладёт лот обратно в пул и переоценивает (alert админу). Игрок видит «выплата отложена, дождитесь стабилизации сети».
5. Audit-запись `PRIZE_PAYOUT` с полным контекстом: `lot_id`, `amount`, `actual_fee`, `tx_hash`, `recipient_address`.

**Жёсткое инвариант:** игрок никогда не подтверждает выплату при недостаточном `fee_buffer`. Это техзадание — `IFeeEstimator` обязан заложить **upper-bound estimate** на момент формирования лота (например, P95 комиссии за прошедшие 7 дней), а не среднее.

#### 12.6.5 KYC / антифрод / лимиты

- **Без KYC** на стороне бота — это TG-бот, не платёжный процессор. Регуляторно — игроки самостоятельно отвечают за свои налоги.
- **Лимит выигрышей на одного игрока** — TODO(balance), но ориентировочно: `max 50 USDT-экв в выплатах за 30 дней`. Сверх — выигрыш зачисляется, но выплата ставится в очередь (visible: «вы достигли месячного лимита, выплата 1.5 USDT придёт после 2026-XX-XX»).
- **Антифрод:** один TG-аккаунт = один TON-адрес для выплат. Смена адреса требует TOTP-подтверждения (если у игрока есть привязка) или 24-часового кулдауна без подтверждения.

#### 12.6.6 Экспозиция в админке

`/balance_*` (Спринт 2.5) расширяется командами:

- `/prize_pool` — текущий баланс пулов (per currency) + список активных лотов + список запроса на выплату.
- `/refund_lot <lot_id>` — вручную вернуть лот в пул (например, при сбое выплаты).
- `/freeze_payouts` — экстренно заморозить все выплаты (например, при подозрении на эксплойт).

Все три — `super_admin`, с TOTP-подтверждением. Audit-запись `ADMIN_PRIZE_*`.

## 18. Система управления нагрузкой (DAU Gate)

### 18.1 Концепция

Администратор устанавливает **лимит DAU** в конфиге бота. Когда количество уникальных активных пользователей за сутки достигает этого лимита — новые регистрации ставятся **в очередь**.

### 18.2 Как работает

```
Конфиг бота:
  MAX_DAU = 500  ← администратор устанавливает и меняет вручную
```

| Ситуация | Действие бота |
|---|---|
| DAU < MAX_DAU | Новые игроки регистрируются **мгновенно** |
| DAU ≥ MAX_DAU | Новый игрок получает сообщение: «Серверы переполнены! Вы в очереди, позиция: #42. Мы уведомим, когда откроется место!» |

### 18.3 Механика очереди

1. **Счётчик DAU** — Redis/in-memory, сбрасывается каждые 24 часа (или rolling window).
2. **Очередь** — FIFO список ожидающих регистрации (PostgreSQL таблица или Redis list).
3. **Разблокировка** — когда DAU падает ниже лимита (вечер/ночь) или администратор увеличивает `MAX_DAU`:
   - Бот автоматически регистрирует следующих в очереди.
   - Отправляет им уведомление: «Ваш пипирик готов! Добро пожаловать!»
4. **Админ-команда** — `/set_max_dau 1000` — мгновенно увеличивает лимит.

### 18.4 Дополнительные возможности

- **Уведомление админу:** Когда DAU достигает 80% от лимита — бот пишет админу: «⚠️ DAU = 400/500 (80%). Рассмотрите увеличение мощностей.»
- **Статистика:** `/admin_stats` — текущий DAU, размер очереди, пик за неделю.
- **VIP-пропуск:** Реферальные игроки или донатеры могут пропускать очередь (опционально).
- **Грейсфул:** Уже зарегистрированные игроки **никогда не блокируются** — лимит только на новые регистрации.

### 18.5 Путь масштабирования

```
Старт:     MAX_DAU = 200   (VPS 1GB)
Рост:      MAX_DAU = 1000  (VPS 2GB)
Масштаб:   MAX_DAU = 5000  (VPS 4GB)
Большой:   MAX_DAU = 20000 (VPS 8GB)
```

Администратор **сам контролирует** когда открывать ворота шире, синхронизируя с апгрейдом сервера.

## 18.6 Админ-панель

> **Основной интерфейс администрирования — Telegram-бот** (команды `/admin_*`). Веб-панель — **дополнительный** инструмент для случаев, когда нужны таблицы, массовые выборки, длинные списки или редактирование конфигов.
>
> Это сознательное решение: «попадать» и «управлять» удобнее через тот же бот, через который играют пользователи. Все админские действия идут через единый код-путь и единый `admin_audit_log`.

### 18.6.1 Цели

- Поддержка игроков (банальные «верните длину», «бот не отвечает»).
- Балансировка экономики на лету (без релиза кода).
- Мониторинг и реагирование на инциденты.
- Антифрод и борьба с накрутками.

### 18.6.0 Канал интерфейса (приоритет)

| Канал | Когда использовать | Приоритет |
|---|---|---|
| **Бот** (`/admin_*`) | Срочные действия: бан, разбан, заморозка, начисление, объявление, мониторинг через `/admin_stats`. Всё, что укладывается в 1–3 параметра. | **Основной** (Фаза 1) |
| **Веб-панель** (FastAPI) | Списки/таблицы/поиск/большие выборки: журнал аудита, история «Главы племени дня», балансовая таблица `display_names`, редактор `balance.yaml`. | Дополнительный (Фаза 2.5+) |

> Любой функционал, реализованный в веб-панели, должен иметь _эквивалент в боте_, если разумно по UX. Команды бота и API панели зовут **один и тот же** use-case в `application/admin/*`.

### 18.6.2 Функционал (по разделам)

| Раздел | Что делает |
|---|---|
| **Дашборд** | Текущие DAU/MAU/concurrent, RPS, очередь регистраций, активные караваны/рейды, ошибки за последний час, статус CI/деплоя. |
| **Игроки** | Поиск по `tg_id` / нику / имени; карточка игрока (длина, толщина, экипировка, титулы); журнал активности; кнопки: бан, разбан, заморозка, +/- длины, +/- толщины (с обязательным указанием причины). |
| **Племена** | Список племён; карточка племени (участники, суммарная длина, история караванов/масс-PvP); просмотр истории «Главы племени дня»; архивация племени; просмотр чата (мета, без истории сообщений). |
| **Активности** | Live-лента активностей (PvP/караван/рейд); возможность принудительно завершить «зависший» караван/рейд с возвратом длины. |
| **Платежи** | Список транзакций (Stars/TON/USDT); статус; ручное «начислить как платёж» (с аудитом); возврат при сбое. |
| **Баланс** | Редактор `balance.yaml` с валидацией и историей версий; «hot-reload» по кнопке; rollback на предыдущую версию. |
| **Локализация** | Просмотр и правка строк локалей (RU/EN/…); статус покрытия каждого языка. |
| **Шаблоны** | Управление JSON-шаблонами логов, предсказаний и пацанских цитат. Добавление новых, отключение, A/B. |
| **Очередь регистраций** | Просмотр очереди, ручное «впустить N человек», изменение `MAX_DAU`. |
| **Анти-фрод** | Подозрительные паттерны рефералов, мульти-аккаунты по `tg_id`/IP/устройству; инструменты массовых действий. |
| **Аудит-лог** | Поиск/фильтрация по `audit_log`: все изменения длины/толщины/предметов/платежей с причиной и автором (бот / администратор / система). |
| **Админы и роли** | RBAC: роли `super_admin`, `support`, `economist`, `read_only`. Список администраторов, выдача/отзыв доступа, журнал входов. |
| **Уведомления** | Алёрты при инцидентах (DAU > 80 %, рост ошибок, сбой платежей). |

### 18.6.3 Требования

**Бот-канал (основной):**
- Команды `/admin_*` доступны только пользователям из таблицы `admins`; проверка по `tg_id` + RBAC-роль.
- Все опасные команды требуют параметра `reason` (минимум 10 символов).
- Все вызовы пишутся в `admin_audit_log` (`admin_id`, `action`, `target`, `before`, `after`, `reason`, `source="bot"`).
- Опасные роли (`economist`, `super_admin`) — обязательная вторая фактор-подтверждающая команда (TOTP-код в виде `/confirm <6 цифр>` после команды-намерения).
- Идемпотентность — каждая команда принимает `idempotency_key` или генерирует его сама из `(admin_id, command, target, timestamp_minute)`.

**Веб-канал (дополнительный):**
- Отдельное веб-приложение (FastAPI + минималистичный фронт: Jinja2 / HTMX / Vue — выбор на этапе архитектуры).
- **Аутентификация**: вход через Telegram Login Widget **обязателен**; пароли — только как fallback с MFA.
- **RBAC**: каждое действие проверяется по роли; `read_only` не может ничего изменить.
- **Аудит-лог админских действий**: каждое изменение пишется в `admin_audit_log` (`source="web"`).
- **2FA** (TOTP) обязателен для ролей `super_admin` и `economist`.
- Все опасные операции требуют **причины** (свободный текст, валидируется минимум 10 символов).
- Идемпотентность ручных начислений — каждое действие имеет `idempotency_key`.
- Сетевой доступ к панели — только по белым IP / VPN / SSH-tunnel; никаких публичных деплоев без auth.
- Логирование всех админских запросов (включая просмотр карточек игроков — для compliance).

### 18.6.4 Безопасность (отдельный приоритет)

| Требование | Реализация |
|---|---|
| HTTPS обязателен | TLS-терминатор (Caddy/Nginx + Let's Encrypt) — для опциональной веб-панели. |
| Защита от CSRF/XSS/SSRF/SQLi | Стандартные практики FastAPI + строгая валидация pydantic. |
| Brute-force | Rate-limit на логин, captcha после 3 неудач, lock на 15 мин. |
| Сегрегация | Админ-панель и бот — **разные процессы**, разные токены БД (минимально необходимые права). |
| Read-only по умолчанию | Любой новый админ начинает с `read_only`, повышение прав — только через `super_admin`. |

#### Bootstrap первого `super_admin`-а

- Первый `super_admin` **не вписывается в код или `balance.yaml`**. Он берётся из env-переменной `BOOTSTRAP_ADMIN_IDS` (список `tg_id` через запятую, обычно один).
- При старте бота: если таблица `admins` пуста и `BOOTSTRAP_ADMIN_IDS` задан — каждому из перечисленных `tg_id` создаётся запись с ролью `super_admin` (одноразово). Если таблица не пуста — переменная **игнорируется**.
- Значение секрета хранится в Devin Secrets (`PIPIRIK_BOOTSTRAP_ADMIN_TG_ID`, `save_scope: org`) и при деплое прокидывается как env. В git/конфиг/логи никогда не попадает.
- Дальше управление админами — через `/admin_grant_role`, `/admin_revoke_role` (только из-под `super_admin`).

### 18.6.5 Команды бота для админов (основной канал)

Полный набор команд для управления через бота (это _основной_ интерфейс админа, см. §18.6.0):

**Поддержка и пользователи:**

| Команда | Роль | Действие |
|---|---|---|
| `/admin_stats` | любая админ-роль | Краткая статистика (DAU, очередь, ошибки за 1 ч, активные караваны/рейды). |
