## 0. Политика разработки (обязательно)

> Эти требования имеют приоритет над любыми соображениями скорости/удобства. Это политика компании и одна из главных инструкций для ИИ-разработчиков и людей-разработчиков, работающих над проектом.

### 0.1 Принципы кода

Проект должен быть **эталонным с точки зрения ООП и SOLID**. Каждое архитектурное решение и каждый PR должны быть проверяемы по этим критериям:

| Принцип | Что значит для проекта |
|---|---|
| **S — Single Responsibility** | Один класс/модуль = одна причина для изменения. Хэндлеры aiogram только парсят апдейт; вся бизнес-логика — в `core/`. |
| **O — Open/Closed** | Новые активности (рейды, караваны, ивенты) добавляются через расширение базовых интерфейсов, без правок старого кода. |
| **L — Liskov Substitution** | Любая реализация интерфейса (например, `ICache`, `IPaymentProvider`) должна быть взаимозаменяемой без изменения вызывающего кода. |
| **I — Interface Segregation** | Узкие, целевые интерфейсы. Никаких «толстых» абстракций «на все случаи жизни». |
| **D — Dependency Inversion** | `core/` зависит только от абстракций. Все конкретные реализации (БД, кэш, Telegram API, платёжные провайдеры) внедряются через DI. |

Дополнительно обязательно:

- **Encapsulation** — внутреннее состояние игрока/клана/каравана меняется только через публичные методы доменных объектов; никаких «прямых правок полей» из хэндлеров.
- **Composition over inheritance** — наследование только когда оправдано (`is-a`); во всех остальных случаях — композиция и стратегии.
- **DRY / YAGNI / KISS** — не плодить дубли, не добавлять «на всякий случай», предпочитать простые решения.
- **Чистая архитектура** — слои `domain → application → infrastructure → bot`, направление зависимостей строго внутрь.
- **Тип-чек обязателен** — `mypy --strict` (или эквивалент) для всего `core/`. Никаких `Any`, `getattr`/`setattr` для обхода типов.

### 0.2 Безопасность и целостность данных (приоритет №1)

Безопасность и целостность данных — **главный нефункциональный приоритет**. Любая фича, которая угрожает им, не принимается, даже если работает.

| Область | Требования |
|---|---|
| **Транзакции** | Все операции, меняющие длину/толщину/инвентарь/членство в кланах — в транзакциях БД. Для критичных операций — уровень изоляции `SERIALIZABLE` или явные блокировки `SELECT ... FOR UPDATE`. |
| **Идемпотентность** | Каждая «опасная» операция (начисление длины, выдача предмета, награда за караван/рейд, платёж) имеет idempotency-key. Повторное выполнение безопасно. |
| **Audit log** | Любое изменение длины/толщины/предметов/платежей пишется в неизменяемую таблицу `audit_log` с причиной, источником и idempotency-key. |
| **Валидация ввода** | Все пользовательские данные валидируются через pydantic-схемы на границе bot ↔ core. Никаких сырых строк глубже хэндлера. |
| **Authz** | Каждая команда проверяет: уровень, длину, членство в клане, «занят ли активностью». Проверки вынесены в декораторы/middleware и покрыты тестами. |
| **Анти-race-conditions** | Activity Lock + транзакции; невозможно одновременно начать два похода / получить две награды за один караван / влиять на чужой профиль. |
| **Секреты** | Только из переменных окружения / vault. Никогда не коммитятся, не логируются, не уходят в сообщения пользователю. |
| **Платежи** | Чек/инвойс хранится в БД до начисления; начисление — только после успешного подтверждения провайдера; idempotency по `payment_id`. |
| **Анти-фрод** | Лимиты команд, детекция накруток рефералов, проверка дублей tg_id, ограничение на количество кланов у одного игрока. |
| **Резервные копии** | Ежесуточный дамп managed PostgreSQL + point-in-time recovery (если есть). Тест восстановления — раз в квартал. |
| **PII** | Хранится минимум: `tg_id`, `language_code`, никнейм TG только в кэше отображения. Запрос на удаление — поддерживается. |
| **Зависимости** | Регулярный аудит (`pip-audit`/`safety`) и обновление до патч-версий. CI блокирует high/critical уязвимости. |

### 0.3 Качество и процессы

- Каждая задача завершается **тестами**: unit + integration + (по возможности) e2e.
- Покрытие `core/*` тестами — **≥ 80 %** (для MVP было 70 %, политика повышена).
- Code review обязателен. Никаких прямых пушей в `main`.
- CI должен проходить: `ruff`, `mypy --strict`, `pytest`, `pip-audit`.
- Любое отклонение от §0.1/§0.2 фиксируется как технический долг с дедлайном.

## 1. Концепция

**Пипирик Варс** — PvP/PvE чат-игра для Telegram, где игроки растят своего «пипирика», сражаются друг с другом, ходят в походы, водят караваны, участвуют в рейдах и объединяются в кланы.

### 1.1 Регистрация игрока и клана

| Сущность | Как добавляется |
|---|---|
| **Игрок** | Пользователь сам обращается к боту в **личных сообщениях** (`/start`). Никаких других способов нет. Если бот видит юзера в группе, но юзер не писал боту в ЛС — играть нельзя. |
| **Клан (= группа TG)** | Любой администратор группы **добавляет бота в группу/супергруппу** и даёт ему права на чтение сообщений и отправку. Бот сам создаёт запись `clans` по `tg_chat_id` и пишет в чат «привет, я зарегистрировал клан…». |
| **Клан (= канал)** | **Не поддерживается** — чтения/реакций пользователей в каналах недостаточно для геймплея, и геймдиз решил отказаться от этой идеи полностью. **Канал-анонсы** (отдельный публичный канал бота с итогами недели / релиз-нотами / лидербордами) — другая фича, она заведена в `current_tasks.md` как backlog-пункт **«канал-анонсы перед публичным релизом»** (отдельный спринт в самом конце Фазы 4). |
| **Привязка игрока к клану** | Игрок становится членом клана автоматически, как только бот видит его сообщения в чате клана **И** игрок зарегистрирован в боте через ЛС (см. выше). До регистрации в ЛС бот его игнорирует и шлёт инструкцию «напишите мне в личку, чтобы вступить». |

> **Важно:** клан **нельзя** «зарегистрировать» через ЛС бота. Это сделано специально — клан = группа в TG, поэтому единственный способ его создать — добавить бота в группу. Это исключает фейковые кланы.

> Бот корректно обрабатывает: смену названия группы, превращение группы в супергруппу (новый `chat_id`), удаление бота из группы (клан **замораживается** — статус `frozen`, не удаляется).

#### Стартовые параметры новичка

| Параметр | Значение |
|---|---|
| **Длина** | 2 см |
| **Толщина** | 1 |
| **Титул** | _нет_ (первый титул «Новичок» выдаётся при первом походе в лес) |
| **Название** | определяется по длине автоматически (см. §2.3) |
| **Имя** | _нет_ (выбивается дропом из леса как предмет — см. §2.5) |

> При регистрации новичок отображается без титула и без имени, например: «Пипирик» (только название). После первого леса станет, к примеру: «Новичок Пипирик Иванушка».

### 2.1 Формат имени (единый, везде)

Во всех сообщениях, логах, топах, боях — имя игрока отображается в формате:

- **Нежный Пипирик Иванушка**

> Титул — за действия (выбирается), название зависит от длины (автоматическое), имя — выбивается в походах.

### 2.3 Название (автоматическое, по длине)

**Полная таблица «диапазон длины → название»** хранится отдельно и редактируется через админ-панель. На уровне ГДД фиксируется только сам факт того, что:

- Каждой длине соответствует ровно одно название.
- Диапазоны не пересекаются и покрывают **всю числовую ось** (от 0 до «бесконечности», в т. ч. отрицательные значения теоретически — но на практике игрок не может уйти ниже 1 см).
- Изменение таблицы — **горячее**, без перезапуска бота.

**Файл-источник истины:** `config/balance.yaml`, секция `display_names`. Формат:

```yaml
# Полуоткрытый интервал: from <= length < to. Последний to = null = «до бесконечности».
display_names:
  - { from: 0,    to: 10,   name: "Пипирик" }
  - { from: 10,   to: 30,   name: "Писюнчик" }
  - { from: 30,   to: 60,   name: "Бананчик" }
  - { from: 60,   to: 100,  name: "Дубинка" }
  - { from: 100,  to: 200,  name: "Жезл" }
  - { from: 200,  to: 500,  name: "Батон по ГОСТу" }
  - { from: 500,  to: null, name: "Батон-переросток" }  # TODO(gd): полная пересборка
```

> **Текущая таблица — временная и подлежит полной перезаписи.** Геймдиз планирует переработать как сами названия, так и границы диапазонов (включая 501+ см и более ранние). До предоставления финального списка используются значения-заглушки выше.
>
> **Валидация:** при загрузке `balance.yaml` бот проверяет, что диапазоны (полуоткрытые: `from ≤ length < to`) покрывают всю шкалу без дыр и пересечений. Если нет — отказ загрузки и алёрт в админ-чат.

## 3. Минимальная длина (порог безопасности)

### 3.1 Правило 20 см

Если у игрока длина **< 20 см**, он **не может** участвовать в активностях, где есть риск потери длины.

| Активность | Может давать −длину? | Требует ≥ 20 см? |
|---|---|---|
| 🌲 Лес | ❌ Нет, **всегда +** | ❌ Нет, доступен всегда |
| 🔮 Предсказание | ❌ Нет, **всегда +** | ❌ Нет, доступно всегда |
| ⛰️ Горы | ✅ Да | ✅ Да |
| 🏰 Данжон | ✅ Да | ✅ Да |
| ⚔️ PvP 1×1 | ✅ Да | ✅ Да |
| ⚔️ Масс PvP | ✅ Да | ✅ Да |
| 🐪 Караван (караванщик) | ✅ Да | ✅ Да, причём **после вычета взноса** ≥ 20 см |
| 🐪 Караван (защитник) | ✅ Да | ✅ Да |
| 🐪 Караван (рейдер, ур. 5+) | ✅ Да | ✅ Да |
| 🐉 Рейд-босс | ✅ Да | ✅ Да |

> **Караван:** Если взнос 15 см, а у игрока 34 см, то после вычета останется 19 см (< 20) — **не может** присоединиться. Нужно минимум 35 см.

### 3.2 Безопасная зона

Игрок с < 20 см **всегда может**:
- Ходить в **лес** (всегда даёт + длину).
- Использовать **предсказание** (всегда даёт + длину).

> **Прокачка толщины** тоже требует, чтобы после вычета стоимости осталось **≥ 20 см**. Например: у игрока 50 см, прокачка стоит 35 см → останется 15 см (< 20) → **нельзя**. Нужно минимум 55 см.

Это гарантирует, что игрок **никогда не застрянет** — всегда может набрать длину через лес и предсказания.

## 5. Блокировка активностей

Игрок может участвовать **только в одной активности** одновременно.

| Активность | Блокирует другие? |
|---|---|
| Поход (лес/горы/данжон) | ✅ Да, на время похода |
| PvP 1×1 | ✅ Да, на время боя |
| Масс PvP | ✅ Да, на время боя |
| Караван | ✅ Да, на 60 минут |
| Рейд-босс | ✅ Да, на время рейда |
| Предсказание | ❌ Нет, мгновенное |

## 6. Кланы (Группы Telegram)

- Каждая **группа Telegram** = **клан**.
- Игрок может состоять в **нескольких кланах**.
- Если игрок в обоих враждующих кланах — **пропускает** масс PvP.
- **Топ кланов** — суммарная длина.

### 6.1 Глава клана дня 👑

Каждые сутки в каждом клане автоматически выбирается случайный «глава клана дня» — для атмосферы и небольшого юмора в чате.

#### Правила:

| Параметр | Значение |
|---|---|
| **Активация** | Клан имеет **≥ 5 активных участников** (зарегистрированных в боте через ЛС). |
| **Лимит частоты** | **Раз в сутки** (по `Europe/Moscow`, как и `/oracle`); сутки сбрасываются в 00:00 МСК. |
| **Триггер — гибридный** (см. ниже) | Кнопка «🎲 Назначить главу дня» в чате клана **ИЛИ** автоматический фоновый розыгрыш в случайное время суток. Что наступит первым — то и побеждает. |
| **Кто выбирается** | Случайный участник клана из активных за последние 7 дней. |
| **Срок полномочий** | До следующего успешного розыгрыша (≤ 24 ч). |
| **Бонус главе дня** | **`uniform(1, 20)` см** длины (балансируемо в `balance.yaml`). Никаких боевых бонусов. |
| **Можно ли выбрать одного и того же** | Можно, но если в клане ≥ 10 участников — алгоритм избегает повтора последних 3 победителей. |
| **Совмещение с другими активностями** | Не блокирует ничего; это «титул дня», а не активность. |

#### Гибридный триггер (важно):

Чтобы не создавать «пиковую нагрузку» в 12:00 МСК (когда тысячи кланов одновременно шлют сообщения), розыгрыш реализован двумя способами одновременно:

1. **🎲 Кнопка / команда `/clan_head` в чате клана.** Любой участник может нажать в любое время. Если на сегодня ещё нет назначенного главы — бот тут же выбирает и публикует. Если уже есть — бот тихо ответит подсказкой («сегодня уже назначен, следующий — после полуночи МСК»).
2. **Фоновый APScheduler-cron.** Для каждого клана при создании/активации генерируется детерминированный персональный offset `0..24 ч` (на основе `clan_id` + `date`). В этот момент cron сам тыкает розыгрыш, если кто-то ещё не сделал это через кнопку.

**Идемпотентность:** уникальный ключ `(clan_id, moscow_date)` в `clan_daily_head`. Параллельные «кнопка + cron» не приведут к двойному назначению.

> Эффект: распределённая нагрузка по суткам + игровой элемент «кто первый дёрнет рулетку». Больше нет необходимости синхронизировать всех на 12:00 UTC.

#### Сообщение в чат клана:

Бот публикует **иронично-смешное** сообщение, заканчивающееся «пацанской цитатой» (в стиле Стэтхема / пабликов ВК / АУФ-цитат). Полное имя игрока — в формате «Титул Название Имя».

```
👑 Главой клана "Огурцы" на сегодня волей рандома (и кнопки от @vasya) назначен:
   Ядрёный Бананчик Коляндр

📏 Получает +14 см к длине за тяжёлое бремя власти и моральные страдания.
🕒 Полномочия — до полуночи по Москве. Потом обратно в строй, как все.

«Пацан сказал — пацан подумал… ну, может быть. Главное — клан стоит, а длина растёт.» 🐺
```

#### Каталог цитат:

- На MVP — **JSON-файл** `templates/clan_quotes_ru.json` / `_en.json` с **≥ 100 цитатами**.
- Цитаты — **ироничные и смешные** (а не пафосные/серьёзные). Стилистика — Стэтхем / паблики ВК / АУФ-цитаты, но с обязательной самоиронией.
- **Уместный мат разрешён** (`mild_profanity = true` в контент-полиси). Цитата с матом помечается тегом `profanity` для опционального фильтра «детский режим клана» в будущем.
- **Запрещено:** политика, межнациональные оскорбления, призывы к насилию, реклама (см. `templates/content_policy.md`).
- Каждая цитата отмечена тегами стиля (`statham`, `vk_pablik`, `auf`, `meme`) и опциональным `profanity` — на будущее для A/B-теста и фильтров.
- Позже — генерация ИИ с проверкой на безопасный контент.

#### Технические требования:

- Реализуется через `application/clan/daily_head/*` use-cases:
  - `RequestDailyHead(clan_id, requested_by)` — синхронный, вызывается из бот-кнопки.
  - `RunDailyHeadCron(clan_id)` — фоновый, идёт от APScheduler-job.
  - Оба использует общий доменный сервис `DailyHeadService.assign_or_get(clan_id, source)` с идемпотентностью.
- Запись в `clan_daily_head` (`clan_id`, `moscow_date` (PK с `clan_id`), `user_id`, `quote_id`, `bonus_cm`, `source` ∈ `{button, cron}`, `requested_by` (nullable)).
- **Идемпотентность**: повторный вызов за те же сутки = no-op (PK = `clan_id, moscow_date`); параллельные кнопка+cron не дают двойного назначения.
- Все начисления длины — через стандартный сервис `progression.add_length(reason="daily_head")` с записью в `audit_log`.
- Если бот удалён из клана / клан в `frozen` или `archived` — оба триггера пропускают этот клан.

> Идея: дополнительная мини-фича — глава дня может **раздать** свой бонус другому участнику клана (опционально, в Фазе 2+).

## 7. PvP-система

### 7.1 PvP 1×1 (с уровня 2, длина ≥ 20 см)

#### Система вызовов:

| Тип вызова | Как работает |
|---|---|
| **Чат → Глобал** (по умолчанию) | Вызов в чате с кнопкой «Принять». **Первые 3 мин** — только участники чата. **После 3 мин** — автоматически в глобальное лобби. |
| **Только чат** | Остаётся только в чате. Никто не принял — отмена. |
| **Только глобал** | Сразу в глобальное лобби. В чате не отображается. |

#### Механика боя:
- 6 кнопок: 3 атаки + 3 блока.
- Каждый выбирает **1 атаку + 1 блок** (через ЛС боту).
- **3 раунда**.
- Попал не в блок → защитник теряет длину. В блок → 0.
- Забавные комментарии после каждого раунда.

### 7.2 Масс PvP (Клановые войны) — все уровни, длина ≥ 20 см

- Клан атакует **не чаще 1 раза в 6 часов**.
- Все участники с длиной ≥ 20 см, **независимо от уровня**.
- Игроки в обоих кланах — пропускают.

**Механика N vs M:**
- Каждый выбирает 1 атаку + 1 блок.
- Каждый бьёт 1 случайного противника.
- Попал не в блок → забирает часть длины. В блок → по нулям.

## 8. Походы (PvE)

### 8.1 Локации

| Локация | Уровень | Длина | Кулдаун (рандом) | Результат | Предметы |
|---|---|---|---|---|---|
| 🌲 Лес | 1+ | Любая | 10–20 мин | **Всегда +** длина | 0–1 шт |
| ⛰️ Горы | 3+ | ≥ 20 см | 20–40 мин | **±** длина | 0–1 шт |
| 🏰 Данжон | 6+ | ≥ 20 см | 40–60 мин | **±** длина | 0–3 шт |

### 8.2 Механика

1. Игрок выбирает поход → проверка уровня, длины и кулдауна.
2. **Сообщение в чат:** «🌲 Ядрёный Бананчик Коляндр ушёл в лес на 14 минут...»
3. Кулдаун тикает. Игрок **заблокирован** от других активностей.
4. По возвращении — **новое сообщение**:

```
🌲 Ядрёный Бананчик Коляндр вернулся из леса!
📏 Длина: +5 см (было 47, стало 52)
🎩 Нашёл: Панамка Новичка [обычный]
   → Надеть | Выбросить
```

- Лес — **всегда + длина** (безопасная активность).
- Горы/Данжон — **±** длина (рискованные).
- Имя = предмет, может выпасть.
- Шмот не копится: надеть или выбросить.

#### Прибавка длины в лесу

Лес имеет **3 ветки исходов** с разной частотой и разным диапазоном прибавки. Распределение веса между ветками — в `balance.yaml`. **Все три ветки — положительные** (лес всегда +).

| Ветка | Диапазон (см) | Стартовый вес | Описание (для логов) |
|---|---|---|---|
| 🥬 Скудный лес | **1–10** | 50 % | «Нашёл +N см под кустом» |
| 🌳 Обычный лес | **5–15** | 35 % | «Зацепился за корягу и нашёл +N см» |
| 🌲 Тучный лес | **10–20** | 15 % | «Наткнулся на полянку и поднял +N см» |

```yaml
forest:
  outcomes:
    - { name: "scarce",   weight: 50, min: 1,  max: 10 }
    - { name: "normal",   weight: 35, min: 5,  max: 15 }
    - { name: "abundant", weight: 15, min: 10, max: 20 }
```

> Точное распределение и границы будут перебалансированы по результатам бета-теста. Запрещено вводить ветку с отрицательной прибавкой — лес остаётся безопасной зоной (см. §3.1).

#### Первый поход → титул «Новичок»

При **первом успешном возвращении из леса** игрок автоматически получает титул **«Новичок»** (`title=newbie`). До этого момента у игрока **нет титула** (см. §1.1, §2.4). Идемпотентно: повторное «первое посещение» (если случается из-за рассинхрона очереди) не выдаёт титул повторно.

## 9. Караваны (создание с уровня 7, нападение с уровня 5)

### 9.1 Концепция

Караван — групповая перевозка длины из клана в клан. Высокий риск + высокая награда. На караван могут напасть рейдеры, а клан-получатель может выставить защитников.

### 9.2 Создание каравана

- **Создатель (лидер):** уровень **7+**, длина после вычета взноса **≥ 20 см**.
- Лидер задаёт:
  - **Вклад длины** — сколько см каждый участник вкладывает.
  - **Пункт назначения** — клан-получатель. Ввести вручную или выбрать из **5 рандомных**.
- Караван создаётся **внутри клана** лидера.

### 9.3 Сбор лобби (20 минут)

- Объявление в чате клана: «🐪 Ядрёный Бананчик Коляндр собирает караван в клан "Батоны"! Вклад: 15 см. Присоединиться?»
- Кнопка **«Присоединиться»** — любой участник клана может вступить, если длина после вычета взноса **≥ 20 см**.
- **20 минут** на сбор. Минимум — **1** (лидер).
- **Кулдаун:** клан может создать караван **не более 1 раза в 12 часов**.

### 9.4 Роли и ограничения для игроков в обоих кланах

| Состоит в... | Караванщик | Защитник | Рейдер |
|---|---|---|---|
| Только в клане-отправителе | ✅ | ❌ | ❌ |
| Только в клане-получателе | ❌ | ✅ | ❌ |
| **В обоих кланах** | ✅ | ✅ | ❌ |
| **Ни в одном из кланов** | ❌ | ❌ | ✅ |
| В одном из двух кланов | — | — | ❌ |

> **Рейдером может быть только тот, кто НЕ состоит ни в клане-отправителе, ни в клане-получателе.**
>
> Игрок в обоих кланах может выбрать **одну** роль: караванщик **или** защитник (не обе одновременно, т.к. блокировка активностей).

### 9.5 Механика каравана

#### Параметры:

| Параметр | Значение |
|---|---|
| Время в пути | **60 минут** |
| Блокировка | Все участники **не могут** участвовать в других активностях |
| Защиты караванщиков | Каждый выбирает **2 блока** (из 3) перед стартом |

#### Рейдеры (нападающие):

- **Уровень 5+**, длина **≥ 20 см**.
- **НЕ состоит** ни в клане-отправителе, ни в клане-получателе.
- Находят караван через **объявление** или **кнопку «Поиск караванов»**.
- Присоединяются **в любое время** в течение 60 минут.
- **Максимум:** ×4 от количества караванщиков (балансируемо).

#### Защитники (клан-получатель):

- Длина **≥ 20 см**. Все уровни.
- Состоит в клане-**получателе**.
- Присоединяются **в любое время** в течение 60 минут.
- Каждый выбирает **1 блок**.
- **Максимум:** ×2 от количества караванщиков.
- Берут **часть ударов на себя**.

#### Боевая механика:

- Каждый рейдер наносит **1 удар** случайному караванщику или защитнику.
- Общее количество ударов = количество рейдеров.
- Макс. ударов = ×4 × караванщиков.

| Ситуация | Результат |
|---|---|
| Рейдер попал **не в блок** | Караванщик/защитник **погибает** + немного **−длины** |
| Рейдер попал **в блок** | Рейдер получает немного **−длины** |

### 9.6 Завершение каравана

#### 🏴‍☠️ Все караванщики и защитники погибли (рейдеры победили):

- Рейдеры **делят всю перевозимую длину поровну** (округление в бо́льшую сторону системой).
- Один случайный рейдер — **Атаман разбойников**: получает **×4 к своей доле** от системы + **титул «Атаман»**.

#### 🐪 Караван дошёл (хотя бы 1 караванщик или защитник выжил):

| Роль | Награда |
|---|---|
| **Лидер каравана** | **×4 к вложенной длине** + **титул «Торговец»** (при 3-м успешном караване) |
| **Караванщики** (остальные) | **×3 к вложенной длине** |
| **Защитники** | **×1 от системы** (фиксированный бонус длины от системы, не от вклада) |
| **Все участники обоих кланов** | **+1 см длины** каждому |

> Все множители (×4 лидер, ×3 караванщики, ×1 защитники, ×4 атаман, +1 клану) — **балансируемые**.

### 9.7 Пример

```
Клан "Огурцы" → Караван в клан "Батоны"
  Лидер: Ядрёный Бананчик Коляндр (вклад 20 см)
  Караванщиков: 5 (вклад 20 см каждый = 100 см общий груз)
  Защитников от "Батонов": до 10
  Рейдеров (не из "Огурцов" и не из "Батонов", ур. 5+): до 20

  Если караван дошёл:
    Лидер:         вложил 20 → получил 80 (×4)
    Караванщики:   вложил 20 → получил 60 (×3)
    Защитники:     получили по X см от системы (×1)
    Все "Огурцы" и "Батоны": +1 см каждому

  Если караван разграблен:
    100 см ÷ рейдеры (поровну)
    1 рейдер = Атаман (×4 к своей доле от системы)
```

## 10. Рейд-боссы

### 10.1 Вызов и участие

| Действие | Уровень | Длина |
|---|---|---|
| **Кинуть вызов** | **9+** | ≥ 20 см |
| **Участвовать** | **4+** | ≥ 20 см |

- Босс = случайный из **топ-30**. Кулдаун вызова: **1 раз в 4 часа** (глобальный).

### 10.2 Управление боссом

- Босс = **сам игрок** (если онлайн) или **бот** (автоматически).

### 10.3 Лобби

- **20 минут** сбор. Кнопка **пересылаемая**. Минимум — **1 игрок**.

### 10.4 Механика

| Роль | Кнопки |
|---|---|
| Босс | 3 атаки → бьёт всех |
| Рейдеры | 3 блока → каждый один |

- Раунды **20 сек – 1 мин**. AFK → бот рандомно.

| Ситуация | Результат |
|---|---|
| Босс попал не в блок | Рейдер выбывает + −длина |
| Босс попал в блок | Рейдер остаётся + забирает длину босса |

### 10.5 Завершение

- **Рейдеры победили** (длина босса < 10 см): делят длину + **% от системы**.
- **Босс победил**: забирает длину рейдеров + **% от системы**.

## 11. Предсказатель

oracle:

  bonus_min: 1

  distribution: "uniform"        # uniform | normal | weighted_buckets

## 13. Социальные фичи

### 13.1 Реферальная система

- Реферальная ссылка: `t.me/pipirik_bot?start=ref_USERID`.
- Бонусы начисляются **после успешной регистрации новичка** (создания записи в `users`):

| Событие | Новичку | Рефереру |
|---|---|---|
| Новичок зарегистрировался | **+5 см** | **+1 см** |
| Новичок достиг толщины **3** | — | **+10 см** |
| Новичок достиг толщины **5** | — | **+30 см** |

- Все начисления — через стандартный `progression.add_length(reason="referral_*")` с записью в `audit_log` и `idempotency_key` (повторное достижение того же уровня не выдаёт бонус повторно).
- **Антифрод:** одна пара `(referrer_id, referred_id)` уникальна; повторные попытки игнорируются. Дополнительные эвристики (одно устройство, IP, поведенческий паттерн) — см. §0.2.
- Конфиг — в `balance.yaml`:

```yaml
referral:
  on_signup:
    new_player_bonus_cm: 5
    referrer_bonus_cm: 1
  on_thickness_milestones:
    - { thickness: 3, referrer_bonus_cm: 10 }
    - { thickness: 5, referrer_bonus_cm: 30 }
```

### 13.2 Шеринг результатов

## 18. Система управления нагрузкой (DAU Gate)

### 18.1 Концепция

Администратор устанавливает **лимит DAU** в конфиге бота. Когда количество уникальных активных пользователей за сутки достигает этого лимита — новые регистрации ставятся **в очередь**.

### 18.2 Как работает

```
Конфиг бота:
  MAX_DAU = 500  ← администратор устанавливает и меняет вручную
```

| Ситуация | Действие бота |
|---|---|
| DAU < MAX_DAU | Новые игроки регистрируются **мгновенно** |
| DAU ≥ MAX_DAU | Новый игрок получает сообщение: «Серверы переполнены! Вы в очереди, позиция: #42. Мы уведомим, когда откроется место!» |

### 18.3 Механика очереди

1. **Счётчик DAU** — Redis/in-memory, сбрасывается каждые 24 часа (или rolling window).
2. **Очередь** — FIFO список ожидающих регистрации (PostgreSQL таблица или Redis list).
3. **Разблокировка** — когда DAU падает ниже лимита (вечер/ночь) или администратор увеличивает `MAX_DAU`:
   - Бот автоматически регистрирует следующих в очереди.
   - Отправляет им уведомление: «Ваш пипирик готов! Добро пожаловать!»
4. **Админ-команда** — `/set_max_dau 1000` — мгновенно увеличивает лимит.

### 18.4 Дополнительные возможности

- **Уведомление админу:** Когда DAU достигает 80% от лимита — бот пишет админу: «⚠️ DAU = 400/500 (80%). Рассмотрите увеличение мощностей.»
- **Статистика:** `/admin_stats` — текущий DAU, размер очереди, пик за неделю.
- **VIP-пропуск:** Реферальные игроки или донатеры могут пропускать очередь (опционально).
- **Грейсфул:** Уже зарегистрированные игроки **никогда не блокируются** — лимит только на новые регистрации.

### 18.5 Путь масштабирования

```
Старт:     MAX_DAU = 200   (VPS 1GB)
Рост:      MAX_DAU = 1000  (VPS 2GB)
Масштаб:   MAX_DAU = 5000  (VPS 4GB)
Большой:   MAX_DAU = 20000 (VPS 8GB)
```

Администратор **сам контролирует** когда открывать ворота шире, синхронизируя с апгрейдом сервера.

## 18.6 Админ-панель

> **Основной интерфейс администрирования — Telegram-бот** (команды `/admin_*`). Веб-панель — **дополнительный** инструмент для случаев, когда нужны таблицы, массовые выборки, длинные списки или редактирование конфигов.
>
> Это сознательное решение: «попадать» и «управлять» удобнее через тот же бот, через который играют пользователи. Все админские действия идут через единый код-путь и единый `admin_audit_log`.

### 18.6.1 Цели

- Поддержка игроков (банальные «верните длину», «бот не отвечает»).
- Балансировка экономики на лету (без релиза кода).
- Мониторинг и реагирование на инциденты.
- Антифрод и борьба с накрутками.

### 18.6.0 Канал интерфейса (приоритет)

| Канал | Когда использовать | Приоритет |
|---|---|---|
| **Бот** (`/admin_*`) | Срочные действия: бан, разбан, заморозка, начисление, объявление, мониторинг через `/admin_stats`. Всё, что укладывается в 1–3 параметра. | **Основной** (Фаза 1) |
| **Веб-панель** (FastAPI) | Списки/таблицы/поиск/большие выборки: журнал аудита, история «Главы клана дня», балансовая таблица `display_names`, редактор `balance.yaml`. | Дополнительный (Фаза 2.5+) |

> Любой функционал, реализованный в веб-панели, должен иметь _эквивалент в боте_, если разумно по UX. Команды бота и API панели зовут **один и тот же** use-case в `application/admin/*`.

### 18.6.2 Функционал (по разделам)

| Раздел | Что делает |
|---|---|
| **Дашборд** | Текущие DAU/MAU/concurrent, RPS, очередь регистраций, активные караваны/рейды, ошибки за последний час, статус CI/деплоя. |
| **Игроки** | Поиск по `tg_id` / нику / имени; карточка игрока (длина, толщина, экипировка, титулы); журнал активности; кнопки: бан, разбан, заморозка, +/- длины, +/- толщины (с обязательным указанием причины). |
| **Кланы** | Список кланов; карточка клана (участники, суммарная длина, история караванов/масс-PvP); просмотр истории «Главы клана дня»; архивация клана; просмотр чата (мета, без истории сообщений). |
| **Активности** | Live-лента активностей (PvP/караван/рейд); возможность принудительно завершить «зависший» караван/рейд с возвратом длины. |
| **Платежи** | Список транзакций (Stars/TON/USDT); статус; ручное «начислить как платёж» (с аудитом); возврат при сбое. |
| **Баланс** | Редактор `balance.yaml` с валидацией и историей версий; «hot-reload» по кнопке; rollback на предыдущую версию. |
| **Локализация** | Просмотр и правка строк локалей (RU/EN/…); статус покрытия каждого языка. |
| **Шаблоны** | Управление JSON-шаблонами логов, предсказаний и пацанских цитат. Добавление новых, отключение, A/B. |
| **Очередь регистраций** | Просмотр очереди, ручное «впустить N человек», изменение `MAX_DAU`. |
| **Анти-фрод** | Подозрительные паттерны рефералов, мульти-аккаунты по `tg_id`/IP/устройству; инструменты массовых действий. |
| **Аудит-лог** | Поиск/фильтрация по `audit_log`: все изменения длины/толщины/предметов/платежей с причиной и автором (бот / администратор / система). |
| **Админы и роли** | RBAC: роли `super_admin`, `support`, `economist`, `read_only`. Список администраторов, выдача/отзыв доступа, журнал входов. |
| **Уведомления** | Алёрты при инцидентах (DAU > 80 %, рост ошибок, сбой платежей). |

### 18.6.3 Требования

**Бот-канал (основной):**
- Команды `/admin_*` доступны только пользователям из таблицы `admins`; проверка по `tg_id` + RBAC-роль.
- Все опасные команды требуют параметра `reason` (минимум 10 символов).
- Все вызовы пишутся в `admin_audit_log` (`admin_id`, `action`, `target`, `before`, `after`, `reason`, `source="bot"`).
- Опасные роли (`economist`, `super_admin`) — обязательная вторая фактор-подтверждающая команда (TOTP-код в виде `/confirm <6 цифр>` после команды-намерения).
- Идемпотентность — каждая команда принимает `idempotency_key` или генерирует его сама из `(admin_id, command, target, timestamp_minute)`.

**Веб-канал (дополнительный):**
- Отдельное веб-приложение (FastAPI + минималистичный фронт: Jinja2 / HTMX / Vue — выбор на этапе архитектуры).
- **Аутентификация**: вход через Telegram Login Widget **обязателен**; пароли — только как fallback с MFA.
- **RBAC**: каждое действие проверяется по роли; `read_only` не может ничего изменить.
- **Аудит-лог админских действий**: каждое изменение пишется в `admin_audit_log` (`source="web"`).
- **2FA** (TOTP) обязателен для ролей `super_admin` и `economist`.
- Все опасные операции требуют **причины** (свободный текст, валидируется минимум 10 символов).
- Идемпотентность ручных начислений — каждое действие имеет `idempotency_key`.
- Сетевой доступ к панели — только по белым IP / VPN / SSH-tunnel; никаких публичных деплоев без auth.
- Логирование всех админских запросов (включая просмотр карточек игроков — для compliance).

### 18.6.4 Безопасность (отдельный приоритет)

| Требование | Реализация |
|---|---|
| HTTPS обязателен | TLS-терминатор (Caddy/Nginx + Let's Encrypt) — для опциональной веб-панели. |
| Защита от CSRF/XSS/SSRF/SQLi | Стандартные практики FastAPI + строгая валидация pydantic. |
| Brute-force | Rate-limit на логин, captcha после 3 неудач, lock на 15 мин. |
| Сегрегация | Админ-панель и бот — **разные процессы**, разные токены БД (минимально необходимые права). |
| Read-only по умолчанию | Любой новый админ начинает с `read_only`, повышение прав — только через `super_admin`. |

#### Bootstrap первого `super_admin`-а

- Первый `super_admin` **не вписывается в код или `balance.yaml`**. Он берётся из env-переменной `BOOTSTRAP_ADMIN_IDS` (список `tg_id` через запятую, обычно один).
- При старте бота: если таблица `admins` пуста и `BOOTSTRAP_ADMIN_IDS` задан — каждому из перечисленных `tg_id` создаётся запись с ролью `super_admin` (одноразово). Если таблица не пуста — переменная **игнорируется**.
- Значение секрета хранится в Devin Secrets (`PIPIRIK_BOOTSTRAP_ADMIN_TG_ID`, `save_scope: org`) и при деплое прокидывается как env. В git/конфиг/логи никогда не попадает.
- Дальше управление админами — через `/admin_grant_role`, `/admin_revoke_role` (только из-под `super_admin`).

Полный набор команд для управления через бота (это _основной_ интерфейс админа, см. §18.6.0):

### Фаза 1 — MVP
1. Регистрация **через ЛС бота** (старт: длина 2, толщина 1, без титула, без имени)
2. Регистрация **клана** через добавление бота в группу/супергруппу; кик бота → `frozen`
3. Правило 20 см (порог безопасности)
4. Система DAU Gate
5. Походы в лес (всегда +, 3 ветки исходов, первое возвращение → титул «Новичок»)
6. Прокачка толщины
7. Шмот и **имена-как-предметы** (дроп из леса, надеть/выбросить)
8. Предсказатель (JSON, всегда +, кулдаун по Москве, +1..+20 см)
9. Топ игроков (формат «Титул Название Имя» с учётом отсутствующих частей)
10. Предгенерированные логи
11. Локализация RU + EN
12. Реферальная система (+5/+1 при регистрации, +10 за тол. 3, +30 за тол. 5)
13. Базовый админ-интерфейс **в боте** (`/admin_stats`, `/find_player`, `/freeze`, `/grant_length` — с TOTP-подтверждением)
