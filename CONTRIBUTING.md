# Контрибьютинг в Pipirik Wars

Этот файл — обязательное чтение перед открытием PR. Если что-то непонятно — смотри `docs/game_design.md` §0 (Политика разработки) и `docs/development_plan.md` §0 (DevOps + чек-листы).

> **Цель:** надёжный, безопасный, поддерживаемый код. SOLID + clean architecture + security-first. CI gate-ов много, и они **обязательны**.

> **Над проектом параллельно работают разные AI-агенты.** Один агент может неожиданно завершить сессию (закончились токены, обрыв, потеря контекста) посередине задачи. Чтобы передача работы между агентами проходила без артефактов, есть отдельная секция [«Протокол передачи работы между агентами»](#-протокол-передачи-работы-между-агентами). Прочитай её при первом подключении и обращайся каждый раз, когда **берёшь работу у другого агента** или **передаёшь свою**.

## 📚 Документация: что где живёт

В проекте 4 живых документа + 1 файл-памятка для безопасной передачи задач. Каждый имеет ровно одну роль; писать одно и то же в нескольких местах **запрещено**.

| Документ | Роль | Когда обновлять |
|---|---|---|
| **`docs/game_design.md`** (ГДД, диздок) | **Опорный документ:** все игровые механики, формулы, баланс, политики, лимиты, UX-сценарии. Источник правды по «что игра делает». | При изменении игровых правил/баланса (с подачи геймдиза/PM). Технические детали имплементации сюда **не пишутся**. |
| **`docs/development_plan.md`** | **Полный план разработки** от старта до релиза: фазы, спринты, задачи, критерии приёмки, операционные руководства завершённых спринтов. Должен соответствовать ГДД. | При планировании новых спринтов; при завершении спринта — добавляется операционная выжимка («как обслуживать то, что построили»). |
| **`docs/current_tasks.md`** | **Только текущий PR** и его статус: активная ветка, шаги, чек-лист, известные блокеры. Перед каждым коммитом — обновить. После мерджа — пересоздать под следующий PR (старое содержимое уехало в `history.md`). | Перед каждым `git commit` (отметить готовые шаги). При смене активного PR-а / ветки. При смене агента (см. протокол ниже). |
| **`docs/history.md`** | **Хронологический журнал** завершённых работ: спринт, дата, сводка, артефакты. Свежие — сверху. | После мерджа PR-а — добавляется новая запись. Записи не редактируются задним числом, только новые сверху. |
| **`AGENT_HANDOFF.md`** *(в корне репо, временный)* | **Safety-net** на случай обрыва сессии агента: расширенный контекст для агента-приёмника (что сделано, что в работе, в каком файле остановился, какие команды запускать дальше). | Создаётся уходящим агентом перед первым `git push` своей работы; удаляется отдельным коммитом перед финальным мерджем PR-а. См. протокол ниже. |
| **`README.md`** | **Внешнее лицо проекта** для GitHub: что это, как поднять локально, как запустить тесты. Минимально, без планов и истории. | При изменении setup/run-инструкций или ссылок на основные доки. |

**Жёсткие правила:**

- **Не плоди новые `.md` в `docs/`.** Если нужна новая страница — обоснуй (новая роль, не покрытая существующими файлами). Если содержимое — ответвление одного из 4 живых документов, добавляй его секцией внутрь.
- **Не дублируй спецификацию.** Если правило игры есть в ГДД, в `development_plan.md` ссылайся на ГДД (`см. game_design.md §X.Y`), а не пересказывай.
- **Не оставляй устаревший контент.** Завершённый спринт в `current_tasks.md` — это артефакт. После мерджа — переноси запись в `history.md` тем же PR-ом или сразу после.
- **Архитектурно-операционные руководства** (как добавить новый source длины, как снять soft-ban, как перезагрузить баланс) — это часть `development_plan.md` (приложения к завершённым спринтам), а **не** ГДД.

## 🛠 Ритуалы работы с документацией

### Перед стартом любой задачи

1. **Прочитать §0 ГДД** (`docs/game_design.md`) — политика разработки SOLID/ООП/безопасность.
2. **Прочитать `current_tasks.md`** — что в работе сейчас, на какой ветке, есть ли `AGENT_HANDOFF.md`.
3. **Прочитать связанный спринт в `development_plan.md`** — критерии приёмки твоего шага.
4. **Прочитать связанную секцию ГДД** — игровая спецификация фичи, которую делаешь.
5. **Поднять окружение** — см. [`README.md`](README.md) «Локальная разработка».

### Перед каждым `git commit`

- [ ] Обновить чек-лист в `current_tasks.md` (отметить выполненные шаги, добавить «начат» если шаг в процессе).
- [ ] Обновить «Последний коммит на ветке» в `current_tasks.md` после `git log` (короткий sha + subject).
- [ ] Если работаешь на feature-ветке и не уверен, что закроешь сессию — обновить `AGENT_HANDOFF.md` (`git status`, текущий файл редактирования, незавершённые шаги).

### Перед открытием PR

- [ ] Убрать `AGENT_HANDOFF.md` отдельным коммитом (`git rm AGENT_HANDOFF.md && git commit -m "chore: remove AGENT_HANDOFF before PR"`).
- [ ] Прогнать локальный CI: `make ci` (lint + typecheck + imports + tests). Должен быть зелёным.
- [ ] Описание PR — по шаблону `.github/pull_request_template.md` (если есть) или ad-hoc:
    - **Summary** — что и зачем.
    - **Risk** — green/yellow/red. Yellow/red требуют объяснения.
    - **Testing checklist** — что ревьюер должен проверить руками.

### После мерджа PR-а

- [ ] Добавить запись в `docs/history.md` (свежие — сверху, формат — см. шапку файла).
- [ ] Очистить/пересоздать `docs/current_tasks.md` под следующий PR (или, если следующий PR — продолжение, обновить «Текущая позиция» и чек-лист).
- [ ] Удалить feature-ветку локально и на origin (если merge — squash, что у нас стандарт).

## 🤝 Протокол передачи работы между агентами

> **Контекст.** Над проектом работают разные AI-агенты. Сессия агента может прерваться неожиданно (токены, инструментарий, обрыв). Этот протокол описывает, как уходящий агент оставляет работу в безопасном состоянии, а приходящий — корректно её подбирает.

### Уходящий агент (особенно при риске обрыва)

1. **Если ещё не запушил коммиты** — закоммить текущее состояние (даже WIP, с префиксом `WIP:` в заголовке, телом коммита поясни в чём ты остановился) и `git push` свою feature-ветку. Доделай, если получится — но **сначала пуш**.
2. **Создай `AGENT_HANDOFF.md` в корне репо** по шаблону ниже. Пуши его отдельным коммитом. Это safety-net: если оборвёшся — следующий агент найдёт всё, что нужно.
3. **Обнови `docs/current_tasks.md`** — секции «Текущая позиция» и «Чек-лист текущего PR» должны быть синхронны с реальным состоянием.
4. **НЕ открывай PR** если ветка в полусломанном состоянии (CI красный, незакончена интеграция). Следующий агент откроет PR сам, когда доведёт до зелёного.

### Шаблон `AGENT_HANDOFF.md`

```markdown
# AGENT HANDOFF — Спринт <X.Y.Z> (шаг N/M)

> Этот файл — временный safety-net. Удали его отдельным коммитом перед открытием PR.

## Что я сделал в этой сессии
- <коммит 1: тема>
- <коммит 2: тема>
- ...

## На каком файле/задаче остановился
- Файл: `src/pipirik_wars/<...>`
- Что планировал дальше: <развёрнутая фраза>
- Где брать ТЗ: `docs/development_plan.md` §<X> / Спринт <Y> / задача <Y.Z>; `docs/game_design.md` §<W>

## Состояние ветки
- Ветка: `devin/<unix_ts>-<slug>`
- База: `main` (или `<другая ветка>`, если PR-цепочка)
- Последний коммит: `<short-sha>` `<subject>`
- Незакоммиченные изменения: <да: список файлов / нет>
- CI прогонялся? <да, зелёный / нет / красный>

## Команды для следующего агента
- Поднять окружение: см. README.md «Локальная разработка»
- Прогнать CI: `make ci`
- Запустить только нужные тесты: `pytest tests/unit/<...> -q`

## Известные блокеры / открытые вопросы
- <если есть, или «нет»>
```

### Приходящий агент (приёмка задачи)

При первом подключении к проекту, **прежде чем что-либо менять**, выполни приёмку. Промпт ниже можно использовать как чек-лист (или скопировать в начало своей сессии).

#### 📝 Промпт-приёмка для нового агента

> *Скопируй этот блок и пройди по нему пункт за пунктом. Это твой первый шаг при подключении.*

```
Я подключаюсь к репозиторию Pipirik Wars и продолжаю работу другого агента.
Прежде чем менять код, я выполню приёмку:

1. Найти HANDOFF.
   - Проверить наличие `AGENT_HANDOFF.md` в корне репо.
   - Если есть — прочитать целиком. Это самый свежий контекст.
   - Если нет — это нормально (агент мог завершить аккуратно). Главный источник
     правды тогда — `docs/current_tasks.md` секция «Снимок состояния проекта»
     и «Текущая позиция».

2. Понять состояние веток.
   - `git fetch origin --prune`
   - `git branch -r | grep -i sprint` — все feature-ветки.
   - `git log --oneline origin/main -10` — что смержено последним.
   - Если HANDOFF указывает feature-ветку — `git checkout <ветка>` и
     `git log --oneline -10` на ней.
   - Если в `current_tasks.md` снимок состояния расходится с реальным
     состоянием git-а — реальное git-состояние приоритетно. Зафиксируй
     расхождение в своих заметках.

3. Прочитать документацию.
   - `docs/game_design.md` §0 (Политика разработки) — обязательно при первом подключении.
   - `docs/development_plan.md` §0 + связанная секция твоего спринта — критерии приёмки.
   - `docs/current_tasks.md` — что должно быть готово сейчас, что — следующее.
   - `docs/history.md` — последние 2–3 записи, чтобы понять контекст.
   - `CONTRIBUTING.md` — этот файл, если ещё не прочитал.

4. Проверить, что код в зелёном состоянии.
   - `make ci` (lint + typecheck + imports + tests). Должен быть зелёным.
   - Если красный — НЕ начинай новую работу. Сначала разбираешься, в каком
     состоянии оставил предыдущий агент, чинишь / откатываешь, доводишь до
     зелёного. Логи падений могут указать, где именно прервалась работа.

5. Удалить артефакты предыдущего агента (после того как взял работу).
   - Если ты будешь продолжать работу того же спринта — `AGENT_HANDOFF.md`
     обнови под себя (или оставь, если он всё ещё актуален).
   - Если уходящий агент закончил спринт, но не успел убрать `AGENT_HANDOFF.md`
     перед открытием PR — удали его отдельным коммитом.
   - Не оставляй два конфликтующих описания текущего состояния
     (`AGENT_HANDOFF.md` + `current_tasks.md` секция «Снимок» с разными
     данными). Один из них приведи в соответствие с реальностью.

6. Обнови `current_tasks.md` под себя.
   - Перепиши секцию «Текущая позиция» с актуальными значениями.
   - Заполни «Чек-лист текущего PR» — что планируешь делать в этой сессии.

7. Только теперь — приступай к коду.
```

### Что считается «артефактами» (и которые надо вычистить)

- `AGENT_HANDOFF.md` оставленный на feature-ветке после открытия PR (должен быть удалён).
- Несколько `AGENT_HANDOFF.md` от разных агентов в истории ветки (только один актуальный должен жить, остальные удаляются коммитом).
- `current_tasks.md` с устаревшим содержимым (например, активный спринт указан 2.2, реально идёт 2.4).
- Расхождение между «Снимок состояния» в `current_tasks.md` и реальным git-логом.
- Незакоммиченные изменения после длительного простоя.
- Дубликаты в документации (одна и та же спецификация в двух файлах) — этот PR-цикл (от 2026-05-05) их вычистил, но контролируй заново при правках.

## Workflow PR-а

1. **Завести feature-branch.** Формат: `devin/<unix_timestamp>-<short-slug>`. Никогда не пушить напрямую в `main`.
2. **Реализовать фичу с тестами.** Покрытие `domain/` + `application/` ≥ 80 %. Без тестов PR не мерджится.
3. **Прогон локального CI:** `make ci`. Должен быть зелёным **до** пуша.
4. **Открыть PR** в `main`. Описание — по шаблону (см. «Перед открытием PR» выше).
5. **CI должен пройти.** Лента CI: lint+types+tests на 3.11 + 3.12, pip-audit (security). Все обязательны.
6. **Code review.** Минимум 1 апрув. Замечания — в новых коммитах (амендить нельзя — см. ниже).
7. **Squash merge.** История `main` остаётся чистой.
8. **После мержа:** см. секцию «После мерджа PR-а» выше.

### Правила git

- **НЕ** делать `git commit --amend` после первого пуша — портит ревью-ленту. Только новые коммиты.
- **НЕ** force-push в `main`. В feature-branch — только `--force-with-lease` после ребейза.
- **НЕ** коммитить `.env`, секреты, dump-ы БД. `.gitignore` это закрывает, но контролируйте `git add` руками.
- **НЕ** ставить `--no-verify` на pre-commit hooks. Если хук падает — пофиксить причину, а не обходить.
- **НЕ** менять `git config` в репо.

## Чек-лист SOLID + ГДД §0

Каждый PR должен проходить:

- [ ] **SRP** — каждый класс/модуль ровно одна ответственность.
- [ ] **OCP** — расширяемость через новые реализации портов, а не через `if isinstance` в core-коде.
- [ ] **LSP** — все реализации портов взаимозаменяемы; контракт описан в Protocol/ABC.
- [ ] **ISP** — порты узкие (read-side / write-side разделены: `IBalanceConfig` vs `IBalanceReloader`).
- [ ] **DIP** — `domain` и `application` зависят только от Protocol-ов; реализации в `infrastructure`.
- [ ] **Тестируемость** — use-case-ы тестируются с FakeUnitOfWork / FakeRandom / FakeClock без БД.
- [ ] **Никакой случайности и времени** в `domain/application` напрямую — только через `IRandom` / `IClock`.

## Чек-лист безопасности

- [ ] **Транзакции** — все state-mutations в `application` идут через `IUnitOfWork.transaction()`. Никаких «частичных» успехов.
- [ ] **Idempotency** — write-path-команды защищены `IIdempotencyService` с осмысленным ключом.
- [ ] **Audit log** — каждое изменение состояния пишет audit-запись через `IAuditLogger`. Никаких «тихих» апдейтов.
- [ ] **RBAC / AuthContext** — admin-only handler-ы провалидированы через `AuthContext`-middleware. Никаких ручных проверок `tg_id == ...`.
- [ ] **Activity lock** — конкурирующие действия на одного игрока (например, повторный `/forest`) защищены `IActivityLockService`.
- [ ] **Rate limit** — публичные команды проходят через `ThrottleMiddleware` (token-bucket).
- [ ] **Никаких секретов в код / git / логи** — только через env. `SecretStr.__repr__` маскирует значение.
- [ ] **Без `Any`, `getattr`, `setattr`** — типы выписаны явно, mypy --strict проходит.

## Definition of Done для любой задачи

- [ ] **SOLID-чек-лист** пройден (см. выше).
- [ ] **Чек-лист безопасности** пройден (см. выше).
- [ ] Юнит-тесты добавлены и зелёные (`pytest`).
- [ ] Покрытие нового кода ≥ 80 %.
- [ ] Линтеры зелёные (`ruff`, `mypy --strict`, `import-linter`).
- [ ] `pip-audit` без high/critical.
- [ ] Логика бизнеса в `domain/` или `application/`, не в `bot/handlers/*`.
- [ ] Все строки сообщений — в `locales/*` (RU + EN).
- [ ] Балансовые значения — в `balance.yaml`, не зашиты в код.
- [ ] Описание изменения добавлено в `history.md` (новая запись сверху).
- [ ] Если задача завершена — её запись удалена из `current_tasks.md` (или раздел переведён в следующий PR).

## CI gates (обязательны)

| Gate | Команда | Что проверяет |
|---|---|---|
| ruff | `make lint` | Стиль (PEP 8, импорты, line length) + базовые баги. |
| mypy --strict | `make typecheck` | Все аннотации обязательны; `Any` запрещён. |
| import-linter | `make imports` | Контракт слоёв clean architecture. |
| pytest | `make test` | Все тесты + coverage ≥ 80 % (`--cov-fail-under=80`). |
| pip-audit | `make audit` | CVE-проверка зависимостей (отдельной job-ой в CI). |

Локально всё разом — `make ci`.

## Структура тестов

```
tests/
├── unit/             # быстрые, in-memory, без сети и БД
│   ├── domain/       # чистая бизнес-логика (entities, services)
│   └── application/  # use-cases с FakeUnitOfWork / FakeRandom / FakeClock
├── integration/      # БД + миграции (через aiosqlite в CI)
├── load/             # параллельные сценарии (race-conditions)
└── fakes/            # in-memory реализации портов
```
